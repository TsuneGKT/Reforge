## 一、系统定位

敌人系统是 Reforge 的战斗节奏来源。敌人不是单纯的扣血单位，而是通过攻击前摇、攻击频率、可弹反窗口、移动压力和组合关系，逼迫玩家在普攻、弹反、闪避、超频动作之间切换。

在核心循环中，敌人承担三件事：

| 职责 | 说明 |
|------|------|
| 提供操作问题 | 通过前摇、追击、远程压制、范围限制等方式制造玩家必须应对的局面 |
| 提供能量机会 | 可弹反攻击让玩家把敌人攻势转化为能量收入 |
| 提供构筑检验 | 不同敌人组合检验玩家当前 Build 是否能处理多种压力 |

如果敌人只会稳定、清晰、可弹反的攻击，闪避、走位和超频会失去价值；如果敌人大量不可弹反，弹反作为前期能量启动器又会断裂。因此敌人生态必须同时保留“可学习的弹反机会”和“必须切换动作的压力”。

---

## 二、设计目标

### 2.1 让敌人教会玩家动作语言

P0 阶段的第一个敌人必须先教会玩家基础战斗语法：

```text
识别前摇 → 选择弹反或闪避 → 成功应对 → 获得输出或能量机会
```

锈犬承担这个职责。它的攻击前摇要清楚，节奏要稳定，伤害要低，弹反窗口要宽。教学怪的目标不是考验玩家，而是让玩家第一次成功理解“敌人的攻击也是我的资源机会”。

### 2.2 让不同动作都有出场理由

敌人设计不能只服务弹反。Reforge 的玩家行为系统包含普攻、弹反、闪避、超频和技能，敌人生态需要分别给这些动作制造出场空间。

| 玩家动作 | 敌人侧提供的理由 |
|----------|------------------|
| 普攻 | 处理低血量群体、攻击后摇窗口、不可弹反目标 |
| 弹反 | 应对前摇清晰、命中点明确的攻击，并获得能量 |
| 闪避 | 应对群体压迫、范围伤害、不可弹反攻击、站位危险 |
| 超频动作 | 在危险局面中快速清怪、保命或扩大反击收益 |
| 技能 | 中后期处理高密度敌群、Boss 阶段压力或 Build 成型后的输出循环 |

### 2.3 让怪物组合产生变化

单个敌人是教学单位，房间里的敌人组合才是战斗体验。前期可以让玩家用单一策略解决问题；中期开始通过组合打破单一答案；后期在 Build 成型后提高密度，让玩家验证构筑强度。

## 三、敌人类型

敌人按“给玩家制造什么问题”分类，而不是只按外观或数值分类。

| 类型 | 设计作用 | 代表敌人 | 主要考验 |
|------|----------|----------|----------|
| 节奏型 | 提供清晰前摇和稳定攻击节奏 | 锈犬、腐鹿、巡墙甲 | 弹反、攻击后摇输出 |
| 压迫型 | 用速度、数量或持续接触压缩安全空间 | 甲鼠、游魂灯 | 闪避、走位、快速清理 |
| 远程型 | 从远处制造蓄力压力或弹道威胁 | 残弩 | 目标优先级、弹反或闪避选择 |
| 区域限制型 | 改变房间安全区域 | 孢菇、铁罐、残烛祭像 | 站位、路线选择、清障 |
| 精英型 | 以复杂行为检验阶段性 Build | 异铸骑士 | 多动作切换、持续输出、资源管理 |

这些类型不是互斥标签。一个敌人可以同时具备多个职能，但 P0/P1 阶段每个敌人应有一个最明确的核心职责，避免设计意图混乱。

## 四、区域推进与怪物生态

怪物按区域对应骑士团驻地从外到内的结构。每个怪物既要有玩法职能，也要有被异力侵蚀后的叙事来源。

| 区域 | 主要敌人方向 | 设计重点 |
|------|--------------|----------|
| 荒野 | 锈犬、腐鹿、孢菇 | 建立弹反、闪避和基础走位意识 |
| 防御工事 | 残弩、铁罐、巡墙甲 | 引入远程威胁、陷阱和稳定近战压力 |
| 骑士团内部 | 甲鼠、游魂灯、残烛祭像 | 提高群体压迫和空间限制 |
| 精英战斗 | 异铸骑士等 | 作为小型阶段考验，检验动作切换与 Build 强度 |

前期怪物应该“清楚地教”；中期怪物应该“逼玩家换动作”；后期怪物应该“让 Build 展现强度”。

## 五、通用结构

敌人实现应遵循 Godot 组合式结构。不同敌人复用基础组件，通过 EnemyData 和状态机差异形成行为区别。

### 5.1 数据结构

敌人基础数值使用 `EnemyData` Resource 管理，避免把调参值写死在脚本中。

通用字段包括：

| 字段 | 说明 |
|------|------|
| `display_name` | 多语言 key |
| `max_health` | 最大生命值 |
| `attack_damage` | 基础攻击伤害 |
| `move_speed` | 移动速度 |
| `detection_range` | 发现玩家距离 |
| `attack_range` | 进入攻击行为的距离 |
| `attack_interval` | 攻击间隔或行为冷却 |
| `stun_duration` | 被弹反或被打断后的硬直时间 |

具体敌人可以扩展专属字段，例如扑击距离、蓄力时间、远程弹道速度、爆炸范围等。这些细节写入对应 `策划案_敌人名`。

### 5.2 场景结构

敌人场景采用可复用组件组合：

| 模块 | 职责 |
|------|------|
| Enemy 根节点 | 承载移动、朝向、状态机入口 |
| HealthComponent | 管理生命、受伤、死亡 |
| Hurtbox | 接收玩家攻击命中 |
| Hitbox | 在攻击有效阶段对玩家造成伤害 |
| StateMachine | 管理巡逻、追击、攻击、硬直、死亡等状态 |
| Sprite / AnimationPlayer | 表现敌人的动作前摇、攻击、受击、死亡 |

P0 可以先做最小结构，但必须保留后续扩展空间：锈犬不应写成只能服务锈犬的孤立脚本，而应成为后续近战敌人的基准实现。

### 5.3 事件与信号

敌人与玩家、能量系统、房间系统之间通过事件或信号解耦。

| 事件 | 用途 |
|------|------|
| 玩家攻击命中敌人 | 敌人扣血，播放受击反馈 |
| 敌人攻击命中玩家 | 玩家受伤或被闪避无敌规避 |
| 玩家弹反成功 | 敌人攻击被取消，敌人进入硬直，能量系统回能 |
| 敌人死亡 | 房间系统统计清怪进度，可能触发结晶点出现 |

敌人不直接修改玩家能量。敌人提供可弹反攻击机会；玩家弹反成功后，由玩家行为系统和能量系统完成回能结算。

## 六、P0 敌人边界

P0 阶段只实现锈犬，目标是验证第一套敌人闭环：

```text
锈犬发现玩家 → 追击 → 后蹲蓄力 → 铁甲扑击 → 玩家弹反/闪避/受伤 → 锈犬硬直或继续追击 → 被击杀 → 房间收到敌人死亡事件
```

P0 必须验证：

- 锈犬能发现玩家并追击。
- 锈犬进入攻击范围后执行前摇清晰的扑击。
- 锈犬扑击有可弹反窗口。
- 玩家弹反成功后，锈犬取消攻击并进入硬直。
- 玩家普通攻击可以击杀锈犬。
- 锈犬死亡后能通知房间或全局系统。

P0 暂不追求：

- 多种敌人共存。
- 复杂编队或波次系统。
- 敌人之间协同 AI。
- 远程弹道、陷阱、区域限制。
- 精英怪多阶段行为。

## 七、与其他系统的关系

### 7.1 与玩家行为系统

敌人攻击节奏决定玩家行为是否有价值。锈犬、腐鹿、巡墙甲等节奏型敌人服务弹反学习；甲鼠、游魂灯、孢菇等压力型敌人逼迫玩家使用闪避、走位或超频动作。详见 [[DESIGN_玩家行为系统#8.3 与怪物设计]]。

### 7.2 与能量系统

敌人是前期能量循环的来源。玩家并不是自动回复能量，而是通过弹反敌人攻击获得能量。敌人设计必须控制可弹反机会的密度：太少会导致能量断流，太多会导致能量过剩。

### 7.3 与关卡结构

房间中的敌人组合决定一场战斗的主要问题。前期单一敌人用于教学，中期混合敌人逼迫动作切换，后期提高密度验证 Build。敌人死亡事件也会影响清房、结晶点和出口门的出现。

### 7.4 与世界观

怪物不是随机生成的怪，而是骑士团遗留物被异力侵蚀后的结果。敌人的外观和行为应保留其原本用途：军犬会巡逻扑击，防御工事会远程射击，教堂圣像会限制区域，骑士残躯会保留战斗本能。

## 九、待验证问题

1. P0 只用锈犬能否充分验证玩家攻击、弹反、闪避、受伤、死亡和能量回收闭环。
2. 锈犬的攻击节奏是否足够清晰，能否让新手在低压力下理解弹反。
3. 敌人可弹反机会密度是否会导致能量过剩或断流。
4. 敌人死亡事件是否足以支撑 P0-5 的清房结晶逻辑。
5. 敌人通用结构是否能扩展到腐鹿、巡墙甲、甲鼠等后续敌人，而不需要重写底层。
