## 1. 设计目的

玩家行为系统的目的不是单纯让角色能移动和攻击，而是建立 Reforge 的基础战斗语法：玩家通过移动寻找位置，通过普通攻击造成稳定输出，通过弹反把敌人攻击转化为能量，通过闪避保留生存空间，再把这些基础动作交给能量、超频和天赋系统继续改写。

如果玩家行为系统只做成普通动作游戏的控制器，弹反、能量和构筑会彼此脱节：玩家会只关心攻击和走位，能量系统无法成为核心循环的入口；如果把弹反做得过硬核，新手又会在 Build 成型前反复失败，构筑体验无法展开。因此 P0 阶段的目标是建立一个“中等容错、弹反可学、动作可扩展”的玩家骨架。

本策划案只定义 P0-2 的玩家基础行为。超频的消耗规则、天赋的改造效果、技能槽的完整玩法不在本案展开，只保留必要入口和信号，避免 P0 范围失控。

---

## 2. 系统定位

玩家行为系统是战斗输入层和状态层，位于能量系统、敌人系统、天赋系统之前。它负责回答三个问题：

1. 玩家当前能不能行动。
2. 玩家当前动作如何与敌人 Hitbox / Hurtbox / ParryArea 交互。
3. 玩家动作成功后向其他系统发出什么事件。

P0 阶段覆盖以下行为：

| 行为 | P0 职责 | 不在 P0 展开 |
|------|---------|--------------|
| 移动 | 八方向移动、碰撞、朝向记录 | 地形特化移动、移动天赋 |
| 普通攻击 | 近战 Hitbox、命中敌人、基础伤害 | 连段树、蓄力攻击、武器差异 |
| 弹反 | 时间窗口、完美/普通弹反、敌人攻击反制 | 弹反流完整天赋、子弹时间、反击派生 |
| 闪避 | 位移、短暂无敌、脱离危险 | 完美闪避、残影、闪避伤害 |
| 受伤/死亡 | 扣血、短暂无敌、死亡信号 | 复活、死亡结算、Meta 惩罚 |
| 超频入口 | 接收开关输入，广播状态 | 具体消耗和强化效果 |
| 技能槽 | 保留接口 | 技能系统实装 |

系统边界：玩家脚本不直接管理能量结晶、天赋池或房间流程。玩家行为只产生事件，例如 `parry_succeeded`、`attack_landed`、`player_hit`、`player_died`。能量增加、HUD 更新、房间清除等由对应系统监听事件处理。

## 3. 具体规则

### 3.1 移动

[基础移动]

触发条件：玩家处于 Idle / Move，且存在移动输入。

序列：
1. 读取 `move_up` / `move_down` / `move_left` / `move_right` 或手柄左摇杆输入。
2. 合成移动方向，归一化后乘以 `move_speed`。
3. 调用 CharacterBody2D 的移动逻辑，与环境墙壁碰撞。
4. 记录最后一次有效输入方向，作为攻击、弹反、闪避的默认朝向。

结束后：输入归零则回到 Idle；输入持续则保持 Move。

设计说明：键盘 WASD 天然接近八方向，手柄左摇杆保留模拟输入。若后续需要更复古的像素动作手感，再增加八方向量化开关，不在 P0 写死。

### 3.2 普通攻击

[普通攻击]

触发条件：玩家处于 Idle / Move，按下 `attack`，且不在受伤、死亡、弹反或闪避状态。

序列：
1. 起手（约 0.08 秒）→ 锁定攻击方向，播放攻击前摇或占位姿态。
2. 判定（约 0.10 秒）→ 激活玩家前方 Hitbox，命中敌人 Hurtbox 时造成 `attack_damage`。
3. 收招（约 0.17 秒）→ 关闭 Hitbox，播放收招或回位动画。

结束后：若仍有移动输入则回到 Move，否则回到 Idle。

规则：
- 初始普通攻击不产生能量。
- 同一次攻击对同一敌人只结算一次命中。
- 命中后发出 `attack_landed(damage)`，供打击反馈、天赋或统计监听。
- P0 阶段暂不做连段树，避免抢走弹反系统的测试焦点。

设计理由：普通攻击是稳定输出手段，不应在开局就成为稳定产能入口。若普攻初始产能，玩家会倾向于主动追砍而非读敌人攻击，弹反驱动的能量循环会被削弱。

### 3.3 弹反

[弹反]

触发条件：玩家处于 Idle / Move，按下 `parry`，且不在受伤、死亡、攻击或闪避状态。

序列：
1. 进入 Parry 状态（0.30 秒）→ 激活 ParryArea，记录弹反开始时间。
2. 完美窗口（0.00-0.15 秒）→ 若检测到可弹反敌人攻击，判定为完美弹反。
3. 普通窗口（0.15-0.30 秒）→ 若检测到可弹反敌人攻击，判定为普通弹反。
4. 窗口结束 → 关闭 ParryArea，播放失败或回位动画。

结束后：成功或窗口结束后，回到 Idle / Move。

成功处理：
- 完美弹反：发出 `parry_succeeded(true)`。
- 普通弹反：发出 `parry_succeeded(false)`。
- 被弹反的敌人攻击应取消当前攻击，进入对应敌人的被弹反处理。
- 能量增加由 [[策划案_能量系统]] 或 RunData 监听 `parry_succeeded` 后处理，玩家行为系统不直接改能量。

失败处理：
- 窗口内没有检测到可弹反攻击，则不产生能量。
- 弹反失败不额外惩罚，但玩家在窗口和收束期间会失去移动自由，形成机会成本。

设计理由：0.30 秒总窗口比硬核动作游戏更宽，服务 P0 的能量循环验证；前 0.15 秒区分完美弹反，给高手操作空间。实现以秒为准，不依赖渲染帧率，帧数只作为 60fps 下的手感换算参考。

### 3.4 闪避

[闪避]

触发条件：玩家处于 Idle / Move，按下 `dodge`，且不在受伤、死亡、攻击或弹反状态。

序列：
1. 起跳（立即）→ 读取当前移动输入；若无输入，则使用最后朝向。
2. 位移（0.20 秒）→ 按 `dodge_speed` 朝目标方向移动。
3. 无敌（约 0.15 秒）→ 玩家 Hurtbox 不接收伤害或忽略敌方 Hitbox。
4. 收束 → 恢复 Hurtbox 正常受击。

结束后：若仍有移动输入则回到 Move，否则回到 Idle。

规则：
- 初始闪避不产生能量。
- P0 阶段不实现完美闪避。
- 闪避不默认取消普通攻击，避免玩家用闪避取消抹掉攻击风险。

设计理由：闪避提供非弹反路线的生存手段。后续甲鼠、陷阱、地面预警和不可弹反攻击都需要闪避存在；但初始闪避不产能，才能保留弹反作为前期能量启动器的地位。

### 3.5 受伤与死亡

[受伤]

触发条件：玩家 Hurtbox 被敌方 Hitbox 命中，且玩家不处于无敌状态。

序列：
1. 扣除伤害 → HealthComponent 更新当前生命值。
2. 发出 `player_hit(damage)`。
3. 播放受伤反馈 → 闪红、轻微击退或短暂停顿归入 P0-7 反馈细化。
4. 进入短暂无敌 → 防止多段 Hitbox 同帧连扣。

结束后：生命值大于 0 则回到 Idle / Move；生命值小于等于 0 则进入 Dead。

[死亡]

触发条件：玩家生命值小于等于 0。

序列：
1. 取消当前动作。
2. 关闭输入响应与 Hitbox / ParryArea。
3. 发出 `player_died`。

结束后：进入死亡流程。P0 阶段只做到死亡状态，复活、结算和 Meta 货币不在本案实现。

### 3.6 超频入口

[超频开关]

触发条件：玩家按下 `overclock`。

序列：
1. 若当前可切换超频，则切换 `is_overclock_active`。
2. 发出 `overclock_toggled(is_active)`。
3. 由能量系统、视觉系统或后续天赋系统监听并处理实际消耗与强化。

结束后：玩家保持原本状态。

设计理由：超频覆盖在普通攻击、闪避、弹反之上，不应在玩家状态机中做成完全独立动作。P0-2 只保留入口，完整规则归入 [[策划案_能量系统]]。

### 3.7 中断优先级

玩家行为的中断规则采用“高优先级动作覆盖低优先级动作”的方式。每个中断必须明确属于三种处理之一：取消当前动作、暂停后恢复、当前动作继续并叠加效果。

中断优先级（从高到低）：

[5-最高] 死亡
- 不可被打断。
- 触发后取消当前一切动作。
- 关闭玩家输入、Hitbox、ParryArea。
- 进入 Dead 状态，播放死亡表现，后续交给死亡流程处理。

[4-高] 受伤 / 硬直
- 默认不可被普攻、移动、弹反、闪避打断。
- 触发后取消当前动作，进入 Stun。
- 硬直结束后回到 Idle / Move。
- “受伤中闪避”可以作为后续天赋解锁项；默认状态下不可用。

[3-中高] 弹反
- 不可被普通输入打断。
- 可以打断普攻和移动。
- 进入 Parry 后开启弹反窗口。
- 弹反窗口内接触可弹反攻击 → 触发弹反成功 → 发出 `parry_succeeded(is_perfect)` → 敌人进入被弹反处理。
- 弹反窗口结束未成功 → 弹反失败 → 进入失败收束。

[3-中高] 闪避
- 不可被普攻打断。
- 可以打断普攻和移动。
- 可以被弹反打断：闪避中按下弹反时，取消剩余位移，立即进入 Parry。
- 无敌帧期间不受伤害。
- 结束后回到 Idle / Move。

[2-中] 超频开关
- 任何非硬直、非死亡状态下可触发。
- 不打断当前动作。
- 处理方式：当前动作继续，叠加超频状态，发出 `overclock_toggled(is_active)`。
- 超频只改变动作的能量经济和强化效果，不改变动作优先级。

[1-低] 普通攻击
- 可以被弹反、闪避、受伤、死亡打断。
- 被打断时取消当前攻击剩余阶段，关闭玩家 Hitbox。
- 清空本次攻击派生计数，不保留后续连击收益。
- 正常结束后回到 Idle / Move。

[0-最低] 移动输入
- 可以被所有动作打断。
- 被打断后直接进入对应动作状态。

设计理由：弹反、闪避和普攻都要成为后续天赋可改写的动作入口，因此优先级不能只服务 P0 的“动作锁定”。普攻允许被弹反和闪避取消，可以形成“进攻中读到敌人前摇 → 立刻转防御”的动作表达；代价是取消本次攻击后续收益，避免无成本安全输出。超频不作为独立动作插入状态机，而是叠加状态，这符合 [[DESIGN_玩家行为系统#5.4 超频入口]] 对超频的定位。

[中断交互] 普攻中按下弹反

条件：玩家处于 Attack 起手、判定或收招阶段，按下 `parry`。

处理：取消攻击剩余阶段 → 关闭玩家 Hitbox → 清空本次攻击派生计数 → 立即进入 Parry。

设计理由：弹反优先级高于普攻，允许玩家在进攻中转入防御判断。清空攻击派生计数是机会成本，表示玩家放弃了当前攻击节奏的后续收益。

[中断交互] 普攻中按下闪避

条件：玩家处于 Attack 起手、判定或收招阶段，按下 `dodge`。

处理：取消攻击剩余阶段 → 关闭玩家 Hitbox → 清空本次攻击派生计数 → 立即进入 Dodge。

设计理由：闪避高于普攻，玩家可以用闪避为进攻失误止损。但取消攻击会损失本次输出节奏，避免普攻变成无风险试探。

[中断交互] 闪避中按下弹反

条件：玩家处于 Dodge 位移阶段，按下 `parry`。

处理：取消闪避剩余位移 → 在当前位置进入 Parry → 闪避无敌只保留到状态切换完成，不额外延长弹反安全时间。

设计理由：允许“闪避取消弹反”是有意设计，它为后续“闪避弹反流”提供基础操作语法。但无敌不应覆盖整个弹反窗口，否则玩家可以用闪避无敌抵消弹反失败风险，弹反的时机判断会被削弱。

[中断交互] 弹反中受击

条件：玩家处于 Parry 状态，敌方攻击命中但不可弹反，或未被 ParryArea 成功识别。

处理：取消 Parry → 进入 Stun → 触发正常受伤流程。

设计理由：弹反不是无敌防御。只有可弹反攻击在窗口内才转化为收益；不可弹反或时机错误仍应承担风险。

[中断交互] 超频状态下按下弹反

条件：`is_overclock_active` 为 true，玩家按下 `parry`。

处理：按普通优先级进入 Parry → 弹反成功时同时触发超频弹反的消耗规则与弹反成功的回能规则 → 具体盈亏由 [[策划案_能量系统]] 管理。

设计理由：超频不改变弹反的状态优先级，只改变弹反的能量经济。这样可以保留“超频弹反到底赚不赚”的数值博弈，同时避免玩家状态机因为超频分裂成两套动作。

[中断交互] 任意状态中死亡

条件：玩家当前生命值小于等于 0。

处理：无论当前处于 Move / Attack / Parry / Dodge / Stun / Overclock，立刻取消当前动作 → 关闭输入、玩家 Hitbox、ParryArea → 进入 Dead → 发出 `player_died`。

设计理由：死亡是最高优先级兜底状态，不能被任何输入、动画、无敌帧或超频状态延后。这样可以避免“死亡后继续攻击命中”“死亡后还能弹反”等状态机错误。

[中断交互] 闪避无敌帧中受击

条件：玩家处于 Dodge，且无敌帧仍有效时接触敌方 Hitbox。

处理：忽略本次伤害 → 不进入 Stun → 不发出 `player_hit` → 继续执行 Dodge。

设计理由：闪避的价值必须清楚可感知。无敌帧内受击如果仍然扣血，会让玩家无法判断是时机错误、碰撞错误，还是系统不可靠。

[中断交互] 弹反窗口结束后敌人 Hitbox 仍重叠

条件：玩家 Parry 窗口已经结束，敌方可弹反 Hitbox 仍与 ParryArea 或玩家 Hurtbox 重叠。

处理：不追溯判定弹反成功；若敌人 Hitbox 仍处于有效伤害阶段且玩家不处于无敌状态，则正常受伤。

设计理由：弹反是时机判断，不是只要按过弹反就保底成功。窗口结束后不追溯成功，可以保证弹反节奏严谨，也便于后续调试窗口长度。

[中断交互] 同一帧普攻命中与玩家受伤

条件：同一物理帧内，玩家 Attack Hitbox 命中敌人，同时玩家 Hurtbox 被敌方 Hitbox 命中。

处理：普攻命中和玩家受伤都结算；若玩家因此死亡，则死亡优先级立即接管，取消攻击剩余阶段并进入 Dead。

设计理由：允许“互换血”能保留动作游戏的自然手感，也避免同帧事件因为执行顺序不同产生不稳定结果。死亡仍然作为最高优先级兜底。

[中断交互] 受伤硬直中输入动作

条件：玩家处于 Stun，期间按下 `attack` / `parry` / `dodge` / `overclock`。

处理：默认忽略输入，不缓存；Stun 结束后玩家需要重新输入动作。“受伤中闪避”可作为天赋解锁项，未解锁时不可用。

设计理由：受伤硬直是受击惩罚的一部分，默认不允许用输入缓存绕过。若后续设计“受伤闪避”天赋，它会成为明确的规则例外，而不是基础能力。

[中断交互] 任意可行动作中按下超频

条件：玩家不处于 Dead / Stun，按下 `overclock`。

处理：当前动作继续 → 切换超频状态 → 发出 `overclock_toggled(is_active)`；若玩家处于 Dead / Stun，则忽略本次超频输入。

设计理由：超频是叠加状态，不是插入状态机的动作。它应当改变当前与后续动作的能量经济和强化效果，而不是打断攻击、弹反或闪避的动画流程。

## 4. 数值与依据

| 属性 | 数值 | 依据 |
|------|------:|------|
| 最大生命值 | 100 | 锈犬 8 伤害约 12 次命中才死亡，给新手学习弹反和闪避的容错 |
| 普攻伤害 | 10 | 锈犬 40HP 约 4 刀击杀，不会太快消失，也不拖慢教学节奏 |
| 攻击速度 | 2 次/秒 | 锈犬被弹反后硬直 1 秒，玩家约可打 2 刀，反馈清晰 |
| 普攻动作锁定 | 约 0.35 秒 | P0 先保证出手有风险，避免无成本连点；后续取消规则通过中断优先级控制 |
| 移动速度 | 200 px/s | P0 房间内足以走位，后续根据房间尺寸和敌人移速校准 |
| 能量池 | 100 | 与 100 能量 = 1 光核一致，便于理解资源换算 |
| 弹反总窗口 | 0.30 秒 | 约 18 帧，P0 先保证弹反能稳定启动能量循环 |
| 完美弹反窗口 | 0.15 秒 | 约 9 帧，用于区分精准操作与普通成功 |
| 完美弹反能量 | 12 | 8-10 次完美弹反可攒满一池，符合核心系统数值框架 |
| 普通弹反能量 | 6 | 完美收益的一半，降低新手挫败但保留高手优势 |
| 闪避速度 | 400 px/s | 约为基础移速 2 倍，能明显脱离近战危险 |
| 闪避持续时间 | 0.20 秒 | 位移距离约 80px，接近锈犬扑击位移量，便于教学期对照 |
| 闪避无敌 | 约 0.15 秒 | 覆盖闪避前半段，避免闪避变成长期安全状态 |
| 受击硬直 | 0.25 秒 | 让受伤有明确惩罚，但不长到打断构筑体验 |
| 受击后无敌 | 0.60 秒 | 防止群怪或多段 Hitbox 连续伤害瞬间秒杀 |

弹反窗口采用秒作为实现单位，不直接用帧计数锁死。帧数只用于 60fps 下的手感描述，便于和动画前摇、敌人命中帧、HitStop 反馈对齐。

## 5. 与其他系统的联系

**与能量系统**：弹反成功是 P0 的主要能量输入。玩家行为系统只判定弹反结果并发出 `parry_succeeded`，能量增加由 RunData / EnergySystem 处理。这样可以避免玩家控制器直接承担资源系统职责。→ [[策划案_能量系统]]

**与锈犬**：锈犬是 P0 第一个弹反教学对象。玩家行为系统的弹反窗口、ParryArea、受伤逻辑必须和锈犬的铁甲扑击、16 帧敌方可弹反窗口、被弹反硬直配合验证。→ [[策划案_锈犬]]

**与天赋系统**：天赋系统后续会改写玩家动作规则，例如“闪避后弹反窗口翻倍”“普攻命中累积标记”“弹反成功后短暂进入超频”。因此玩家行为必须提供清晰的动作事件，而不是把所有逻辑写死在单个脚本里。→ [[策划案_天赋系统]]

**与超频系统**：超频是动作强化覆盖层。P0-2 只处理开关输入和状态广播；具体“普攻命中扣 3-5 能量、闪避消耗 8-12、弹反消耗 15-20”等规则属于能量系统与超频实现。→ [[DESIGN_核心系统设计文档#六、超频机制——能量消耗出口]]

**与打击感反馈**：玩家行为是否成立，取决于玩家能不能读懂动作结果。弹反成功、攻击命中、受伤、闪避都要给 P0-7 的 HitStop、屏震、音效和闪烁提供触发点。→ [[策划案_打击感与反馈]]

## 6. 美术/音效需求

P0 阶段使用占位素材即可，但必须能读懂状态。玩家行为测试不能只看一个方块平移，否则弹反、攻击和闪避时机难以判断。

| 需求 | P0 最低标准 | 说明 |
|------|-------------|------|
| idle | 1-2 帧或静态图 | 玩家静止状态可识别 |
| run | 2-4 帧或移动姿态 | 能看出移动方向和速度 |
| attack | 起手 / 出手 / 收招至少可区分 | 用于对齐 Hitbox 激活时机 |
| parry | 明显防御姿态或闪光 | 玩家要知道窗口何时开始 |
| dodge | 残影、位移姿态或短暂透明 | 玩家要知道闪避已触发 |
| hurt | 闪红或受击姿态 | 玩家要知道受伤已经结算 |
| death | 倒地、消散或隐藏 | P0 可简化 |

音效最低需求：

| 音效 | 用途 |
|------|------|
| 普攻挥动 | 确认玩家输入生效 |
| 攻击命中 | 区分空挥与命中 |
| 弹反成功 | 必须明显区别于普通命中 |
| 受伤 | 提醒玩家损失生命 |
| 闪避 | 确认位移与无敌窗口开始 |

## 7. 程序需求

技术方案遵循 [[INDEX_开发路线图#一、Godot 设计哲学速查]]：场景组合、信号解耦、Resource 管理数值，不做深层继承，不在代码中写死平衡值。

### 7.1 场景结构

玩家场景基于 CharacterBody2D：

```text
Player (CharacterBody2D)
├── CollisionShape2D
├── Sprite2D / AnimatedSprite2D
├── AnimationPlayer
├── StateMachine
│   ├── IdleState
│   ├── MoveState
│   ├── AttackState
│   ├── ParryState
│   ├── DodgeState
│   ├── StunState
│   └── DeadState
├── HealthComponent
├── HitboxComponent / AttackArea (Area2D)
│   └── CollisionShape2D
├── HurtboxComponent / HurtboxArea (Area2D)
│   └── CollisionShape2D
├── ParryArea (Area2D)
│   └── CollisionShape2D
└── EnergySystem（P0-3 接入）
```

说明：
- StateMachine 管理玩家当前状态。
- HealthComponent 只管理生命值、受伤、死亡信号。
- AttackArea 只在普通攻击判定阶段激活。
- ParryArea 只在弹反窗口内激活。
- EnergySystem 可在 P0-3 挂载或接入；P0-2 先保证 `parry_succeeded` 事件存在。

### 7.2 数据结构

使用 `PlayerStats` Resource。基础字段沿用 [[策划案_项目基建#2.4 数据资源类]]，并补充本案需要的动作锁定、闪避无敌、受击硬直和受击后无敌时长字段。

| 字段 | 类型 | 默认值 | 说明 |
|------|------|-------:|------|
| max_health | int | 100 | 最大生命值 |
| attack_damage | int | 10 | 普攻伤害 |
| attack_speed | float | 2.0 | 每秒攻击次数 |
| attack_lock_duration | float | 0.35 | 普攻动作锁定时长 |
| move_speed | float | 200.0 | 像素/秒 |
| max_energy | int | 100 | 能量池上限 |
| parry_window_duration | float | 0.30 | 总弹反窗口，秒 |
| perfect_parry_duration | float | 0.15 | 完美弹反窗口，秒 |
| perfect_parry_energy | int | 12 | 完美弹反收益 |
| normal_parry_energy | int | 6 | 普通弹反收益 |
| dodge_speed | float | 400.0 | 闪避速度 |
| dodge_duration | float | 0.20 | 闪避持续时间 |
| dodge_invincible_duration | float | 0.15 | 闪避无敌时长 |
| hurt_stun_duration | float | 0.25 | 受击硬直时长 |
| hurt_invincible_duration | float | 0.60 | 受击后无敌时长 |

### 7.3 输入需求

沿用 [[策划案_项目基建#2.3 输入配置]]：

| Action | 用途 |
|--------|------|
| move_up / move_down / move_left / move_right | 移动 |
| attack | 普通攻击 |
| parry | 弹反 |
| dodge | 闪避 |
| overclock | 超频开关 |
| interact | 交互，P0-5 使用 |
| pause | 暂停，P1 UI 使用 |

### 7.4 碰撞层需求

| Layer | 用途 |
|-------|------|
| Layer 1 | 环境（墙壁） |
| Layer 2 | 玩家 Body |
| Layer 3 | 敌人 Body / Hurtbox |
| Layer 4 | 玩家 Hitbox |
| Layer 5 | 敌人 Hitbox |
| Layer 6 | 玩家 ParryArea |

玩家 Body 在 Layer 2，检测 Layer 1。玩家 Hitbox 在 Layer 4，检测敌人 Hurtbox / Body。玩家 Hurtbox 检测 Layer 5。玩家 ParryArea 在 Layer 6，检测敌人可弹反 Hitbox。

敌方 Hitbox 需要提供可弹反标记，例如 `is_parryable`。弹反成功时，玩家行为系统不直接修改敌人血量，只把弹反事件交给敌人攻击逻辑或 EventBus 处理。

### 7.5 信号需求

优先复用 EventBus 已有信号：

| 信号 | 参数 | 触发时机 |
|------|------|----------|
| parry_succeeded | is_perfect: bool | 玩家弹反成功 |
| player_hit | damage: int | 玩家受伤 |
| player_died | — | 玩家死亡 |
| attack_landed | damage: int | 普攻命中敌人 |
| overclock_toggled | is_active: bool | 超频开关切换 |

玩家本地节点也可发出更细的内部信号，例如 `state_changed`、`invincibility_started`、`invincibility_ended`，但只有跨系统需要监听的事件才进入 EventBus。

### 7.6 状态机方案

使用节点式状态机：

- 每个状态是 StateMachine 下的一个 Node。
- 每个状态提供 `enter()` / `exit()` / `update(delta)` / `physics_update(delta)`。
- StateMachine 持有 `current_state`，负责切换。
- 输入判断集中在可行动状态中，避免多个状态同时响应同一个按键。

P0 状态最小集合：

| 状态 | 职责 |
|------|------|
| Idle | 等待输入 |
| Move | 移动 |
| Attack | 普攻起手、判定、收招 |
| Parry | 弹反窗口计时与 ParryArea 开关 |
| Dodge | 闪避位移与无敌 |
| Stun | 受伤硬直 |
| Dead | 死亡后关闭输入 |

## 8. 验收标准

- [ ] Player 场景基于 CharacterBody2D，能在测试房间内与墙壁发生碰撞。
- [ ] WASD 可移动，斜向移动速度不会比直线更快。
- [ ] 玩家最后朝向会影响攻击、弹反和闪避方向。
- [ ] 按 `attack` 后只在判定阶段激活玩家 Hitbox。
- [ ] 普攻命中敌人时能造成 `attack_damage`，并发出 `attack_landed`。
- [ ] 按 `parry` 后 ParryArea 只在 0.30 秒窗口内激活。
- [ ] 0.00-0.15 秒内弹反成功判定为完美弹反，发出 `parry_succeeded(true)`。
- [ ] 0.15-0.30 秒内弹反成功判定为普通弹反，发出 `parry_succeeded(false)`。
- [ ] 弹反窗口外被击中会正常受伤。
- [ ] 按 `dodge` 后玩家位移约 0.20 秒，并在前 0.15 秒具备无敌。
- [ ] 闪避期间被敌人 Hitbox 接触不会扣血，无敌结束后会正常受伤。
- [ ] 玩家受伤时扣血并发出 `player_hit(damage)`，进入约 0.25 秒受击硬直，并获得约 0.60 秒受击后无敌。
- [ ] 玩家生命值归零后进入 Dead，关闭输入并发出 `player_died`。
- [ ] Attack / Parry / Dodge / Stun / Dead 之间不会出现多个动作判定同时激活的情况。
- [ ] P0 阶段不实装复杂连招、完美闪避、技能系统或复活系统。

## 9. 待验证问题

1. 0.30 秒弹反总窗口是否过宽，是否会削弱锈犬攻击威胁。
2. 完美弹反 12 能量、普通弹反 6 能量的差距是否足以让玩家追求精准时机。
3. 闪避 80px 左右的位移和约 0.15 秒无敌是否适合 P0 测试房间尺寸。
4. 普攻可被弹反/闪避取消是否过于安全，是否削弱出手风险。
5. 普攻初始不产能是否会让无天赋前期过于单调。
6. 玩家 100HP 的容错是否能支撑新手完成第一次“击杀锈犬 → 结晶 → 祭坛”闭环。
