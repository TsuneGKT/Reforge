# 策划案_项目基建

> 路线图位置 → [[INDEX_开发路线图#P0-1 项目基建 + 占位资源准备]]

## 1. 目的

搭建 Godot 工程基础结构，为后续所有系统开发提供统一的代码规范、数据架构和多语言支持。本任务完成后，项目具备：全局信号总线、当局数据管理、数据驱动的配置体系、三语言支持骨架、可用的占位素材。

### 2.2 全局管理器

两个 Autoload 单例，注册顺序：EventBus → RunData（RunData 依赖 EventBus）。

**EventBus（全局信号总线）**

所有系统间的跨节点通信通过 EventBus 信号解耦，不直接调用其他节点方法。

| 信号 | 参数 | 触发时机 |
|------|------|---------|
| parry_succeeded | is_perfect: bool | 弹反成功 |
| player_hit | damage: int | 玩家受伤 |
| player_died | — | 玩家死亡 |
| attack_landed | damage: int | 玩家攻击命中敌人 |
| energy_changed | current: int, max_energy: int | 能量变化 |
| overclock_toggled | is_active: bool | 超频开关切换 |
| light_core_changed | current: float, max_cores: float | 光核进度变化 |
| crystallization_completed | — | 结晶完成 |
| talent_selected | talent_data: Resource | 天赋被选中 |
| enemy_died | enemy: Node | 敌人死亡 |
| all_enemies_cleared | — | 房间内所有敌人清除 |
| room_cleared | — | 房间通关 |
| boss_phase_changed | phase: int | Boss 阶段切换 |

**RunData（当局运行数据）**

管理单局内的所有运行时数据，提供能量增减、结晶、光核消耗等接口。每局开始时调用 reset_run() 重置。

| 数据 | 类型 | 默认值 | 说明 |
|------|------|-------|------|
| energy | int | 0 | 当前能量 |
| max_energy | int | 100 | 能量池上限 |
| light_core_progress | float | 0.0 | 当前光核进度（0.0 ~ 1.0） |
| light_cores | int | 0 | 已完成的光核数量 |
| max_light_cores | int | 1 | 光核存储上限 |
| acquired_talents | Array[Resource] | [] | 当局已获取天赋列表 |
| rooms_cleared | int | 0 | 已通过房间数 |
| enemies_killed | int | 0 | 击杀敌人数 |
| perfect_parries | int | 0 | 完美弹反次数 |

需要的接口：

| 接口 | 说明 |
|------|------|
| reset_run() | 重置所有当局数据 |
| add_energy(amount) | 增加能量，夹到 0 ~ max_energy，广播 energy_changed |
| consume_energy(amount) → bool | 消耗能量，不足返回 false |
| crystallize() | 当前能量全额转化为光核进度，100 能量 = 1 光核，广播相关信号 |
| spend_light_core() → bool | 花费 1 颗光核，不足返回 false |

### 2.3 输入配置

| Action | 默认绑定 | 说明 |
|--------|---------|------|
| move_up | W / 手柄左摇杆上 | 移动 |
| move_down | S / 手柄左摇杆下 | 移动 |
| move_left | A / 手柄左摇杆左 | 移动 |
| move_right | D / 手柄左摇杆右 | 移动 |
| attack | J / 鼠标左键 / 手柄 X | 普通攻击 |
| parry | K / 鼠标右键 / 手柄 B | 弹反 |
| dodge | Space / 手柄 A | 闪避 |
| overclock | L / Shift / 手柄 Y | 超频开关 |
| interact | E / 手柄 RB | 交互（结晶点、祭坛） |
| pause | Escape / 手柄 Start | 暂停 |

布局逻辑：左手 WASD 移动，右手 JKL 动作，参考格斗/动作游戏经典键位。鼠标左右键作为备选。

手柄逻辑：P0 阶段先支持 Xbox / Switch Pro / DualSense 等标准手柄的通用映射。左摇杆负责角色移动，并保留全向模拟输入，不强制量化为八方向；键盘 WASD 仍天然接近八方向。十字键暂不绑定移动，预留给 UI 交互、菜单选择或后续快捷功能。若后续需要更复古的像素动作手感，再在移动脚本中增加“八方向量化”开关。

手柄震动：P0-1 只预留输入绑定；震动反馈归入 P0-7 基础手感反馈实现，用于弹反成功、受伤、Boss 重击、光核结晶等强反馈事件。

### 2.6 占位素材

P0 阶段需要以下占位素材，风格不需要统一，但不能只停留在“能看出是什么”。玩家与敌人的占位素材至少要能看出当前动作状态，否则攻击、弹反、闪避的时机无法可靠测试。

**精灵**

| 素材 | 说明 |
|------|------|
| 玩家角色 | 俯视角角色，需要 idle / run / attack / parry / dodge / hurt 动画帧；若素材不足，parry / dodge / hurt 可先用姿态帧、闪烁、残影或变色占位 |
| 锈犬 | 比玩家略小的角色，需要 idle / run / 蓄力 / 扑击 / 受击或硬直 动画帧 |
| 结晶点 | 小型发光物体，需要 idle / 可交互高亮 / 转化完成 三种状态 |
| 祭坛 | 略大的交互物，需要 idle / 可交互高亮 / 已使用或触发中 三种状态 |

玩家最低可接受标准是 idle / run / attack 三类动作齐全；弹反、闪避、受伤可以用临时反馈补足。锈犬最低可接受标准是 idle / run / 蓄力 / 扑击，因为蓄力和扑击直接决定弹反窗口能不能被玩家读懂。

交互物最低可接受标准是能区分“未交互 / 可交互 / 已完成或触发中”。P0 不要求复杂动画，但必须让玩家知道自己是否站进交互范围、按键是否生效。

**Tile**

| 素材 | 说明 |
|------|------|
| 地板 | 石板/暗色地面，需要能看清可移动区域 |
| 墙壁 | 和地板有明显区分，需要能看清阻挡边界 |

Tile 最低可接受标准是让玩家一眼分清“能走”和“不能走”。P0 阶段不追求完整 Tile 规则，但墙体碰撞边界必须清楚，否则移动和闪避测试会失真。

**UI**

| 素材 | 说明 |
|------|------|
| 血条/能量条 | 纯色矩形也行，需要能显示当前值、变化和满值状态 |
| 光核 UI | 图标或计数文本均可，需要能显示 0/1、进度或完成状态 |

UI 最低可接受标准是能支撑测试判断：当前生命是否变化、能量是否增加或消耗、光核是否已完成。P0 可以丑，但不能让玩家猜数值状态。

**音效**

| 素材 | 说明 |
|------|------|
| 弹反成功 | 金属碰撞/清脆叮声，必须和普通命中明显区分 |
| 攻击命中 | 钝击声，能确认攻击已经命中 |
| 受伤 | 短促痛感音效，需要有危险感 |
| 交互确认 | 确认音/能量音，能提示操作成功 |
| 敌人死亡 | 消散/碎裂音，能提示战斗结果发生变化 |

音效最低可接受标准是反馈清楚。弹反成功、普通命中、受伤、交互、死亡五类声音不能听起来完全一样，否则 P0-7 的手感判断会失去依据。

**推荐免费资源站**

| 资源站 | 内容 | 网址 |
|--------|------|------|
| itch.io | 2D 俯视角精灵和 Tile | https://itch.io/game-assets/free/tag-top-down |
| Kenney.nl | 高质量免费素材包，CC0 协议 | https://kenney.nl/assets |
| OpenGameArt | 社区贡献的免费游戏素材 | https://opengameart.org |
