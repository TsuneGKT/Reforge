## 1. 设计目的

能量系统的目的，是把玩家的战斗操作转化为“即时生存”和“长期构筑”之间的选择。玩家通过弹反获得能量，可以把能量当场花在超频动作上，也可以在战斗结束后把剩余能量结晶为光核，用于祭坛兑换天赋。

Reforge 的能量同时拥有两个出口：战斗中强化动作，战斗后推进构筑。这个设计让玩家持续面对同一个资源问题：这一点能量现在花掉，还是留到战后变成构筑进度？

P0 阶段的设计重点是验证最小闭环：弹反回能 → 超频动作消耗 → 战后选择结晶或保留 → 光核进度推进。

---

## 2. 系统定位

能量系统是连接玩家行为、超频动作、光核结晶和天赋系统的资源层。它不负责具体动作输入，也不负责天赋选择 UI；它只回答三件事：

1. 当前能量是多少。
2. 某个事件是否应该增加或消耗能量。
3. 战斗结束后，玩家是否把当前能量转化为光核进度。

P0 阶段覆盖以下内容：

| 模块 | P0 职责 | 不在 P0 展开 |
|------|---------|--------------|
| RunData | 保存能量、光核进度、光核数量，提供增减与结晶接口 | 存档、Meta 成长、跨局货币 |
| EnergySystem | 监听弹反成功，把完美/普通弹反转化为能量收入 | 普攻回能、闪避回能、技能回能 |
| OverclockSystem | 识别超频键 + 动作请求，处理 P0 三类超频动作消耗 | 超频锁定、自动强化、复杂流派改写 |
| 光核结晶 | 战后互动结晶点，当前能量转为光核进度 | 多房间路线经济、商店/事件消耗 |
| HUD | 显示能量和光核变化 | 完整 UI 美术、复杂状态提示 |

系统边界：玩家行为系统只发出动作事件或超频动作请求；能量系统负责资源结算；HUD 只监听结果变化，不参与规则判断。

## 3. 具体规则

### 3.1 能量基础规则

[能量池]

触发条件：单局开始或 RunData 重置。

序列：
1. 初始化 `energy = 0`。
2. 初始化 `max_energy = 100`。
3. 初始化光核进度与光核数量。
4. 广播 `energy_changed` 与 `light_core_changed`，让 HUD 刷新初始状态。

规则：
- 当前能量不能低于 0。
- 当前能量不能超过 `max_energy`。
- 所有加减能量都通过 RunData 接口处理，避免多个系统各自改数值。

### 3.2 弹反回能

[弹反回能]

触发条件：玩家行为系统发出 `parry_succeeded(is_perfect)`。

序列：
1. EnergySystem 接收弹反成功事件。
2. 如果 `is_perfect == true`，增加 12 能量。
3. 如果 `is_perfect == false`，增加 6 能量。
4. RunData 更新 `energy` 并广播 `energy_changed(current, max_energy)`。

结束后：玩家当前能量变化，HUD 刷新。

设计理由：P0 阶段由弹反启动能量循环。完美弹反收益更高，奖励精准时机；普通弹反也给收益，降低新手前期失败感。

### 3.3 超频动作请求

[超频动作请求]

触发条件：玩家按住 `overclock` 并开始执行 `attack` / `dodge` / `parry`。

序列：
1. 玩家行为系统记录本次动作 `overclock_requested = true`。
2. Player 向 OverclockSystem 提交动作类型。
3. OverclockSystem 判断当前能量是否足够触发对应超频动作。
4. 能量足够时，动作进入超频版本。
5. 能量不足时，动作照常执行普通版本，不强化、不扣能。

结束后：由具体动作决定是否扣能。

设计理由：超频是动作修饰输入。玩家需要提前承诺“这次动作要强化”，动作开始后不再追溯升级。

### 3.4 超频普攻

[超频普攻]

触发条件：玩家按住 `overclock` 开始普通攻击，且能量至少为 3。

序列：
1. 普攻进入超频版本，播放超频攻击占位反馈。
2. 如果普攻空挥，没有命中敌人，不扣能量。
3. 如果普攻命中敌人，先结算攻击命中，再消耗 3 能量。
4. 能量变化后广播 `energy_changed`。

结束后：回到玩家行为系统的 Attack 结束流程。

设计理由：超频普攻承担快速清怪职责。空挥不扣能量，用于奖励精准操作，并降低玩家试用超频普攻的心理成本。

### 3.5 超频闪避

[超频闪避]

触发条件：玩家按住 `overclock` 开始闪避，且能量至少为 8。

序列：
1. 闪避进入超频版本。
2. 闪避触发时立即消耗 8 能量。
3. 本次闪避获得 P0 占位强化，如位移距离增加、无敌时间略延长或视觉残影。
4. 能量变化后广播 `energy_changed`。

结束后：回到玩家行为系统的 Dodge 结束流程。

设计理由：P0 当前先采用触发即扣能，因为闪避的位移和无敌收益在动作开始时已经发生，规则清楚、实现稳定。

后续复盘：P0-4 敌人攻击与玩家受击判定接入后，需要重新评估超频闪避是否加入兜底机制。例如“闪避期间成功规避一次原本会命中的攻击才扣能”，或“先扣能，若未规避攻击则返还部分能量”。在没有敌人攻击命中链路前，暂不硬改，以免超频闪避变成免费位移。

### 3.6 超频弹反

[超频弹反]

触发条件：玩家按住 `overclock` 开始弹反，且能量至少为 15。

序列：
1. 弹反进入超频请求状态。
2. 弹反失败时，不扣能量，不触发超频效果。
3. 弹反成功时，先结算基础弹反回能。
4. 再消耗 15 能量。
5. 触发超频弹反占位强化，如更高反击伤害、更大硬直或范围反馈。
6. 能量变化后广播 `energy_changed`。

结束后：回到玩家行为系统的 Parry 成功或结束流程。

设计理由：弹反本身仍是能量输入口；超频弹反是在这个基础上额外花能量买高收益反击。基础收益先结算、超频消耗后结算，可以让玩家理解“我仍然弹反成功了，只是这次选择付费强化”。

### 3.7 光核结晶

[光核结晶]

触发条件：战斗房间清空后，玩家与结晶点互动。

序列：
1. 检查当前能量是否大于 0。
2. 检查光核是否未达到存储上限。
3. 将当前能量按 `当前能量 / 最大能量` 转化为光核进度。
4. 当前能量清零。
5. 光核进度达到 1.0 时，增加 1 颗光核。
6. 达到光核存储上限后，额外进度不继续积累。
7. 广播 `energy_changed`、`light_core_changed`，必要时广播 `crystallization_completed`。

结束后：结晶点进入已使用状态，或保持不可互动状态。

规则：
- 玩家不互动结晶点，则不发生结晶，能量保留。
- 光核已满时，结晶点不可互动，或提示“光核已满”。
- P0 即使只有一个测试房间，也必须保留“不结晶”的选择。

设计理由：结晶是战后成长选择，需要玩家通过空间行为主动完成。这样可以减少菜单打断，并让“保留能量进下一场战斗”成为明确选择。

### 3.8 结算顺序

同一个动作同时存在基础收益和超频消耗时，采用：

```text
基础收益先结算 → 超频消耗后结算 → 统一刷新 HUD
```

典型例子：

| 行为 | 基础收益 | 超频消耗 | 净变化 |
|------|---------:|---------:|-------:|
| 超频完美弹反 | +12 | -15 | -3 |
| 超频普通弹反 | +6 | -15 | -9 |

设计理由：结算顺序必须稳定，否则同一次超频弹反在不同实现顺序下可能出现“能量不足无法强化”或“先消耗再回能”的歧义。

## 4. 数值与依据

| 项目 | 数值 | 依据 |
|------|------:|------|
| 能量池上限 | 100 | 与光核兑换比例一致，玩家容易理解 100 能量 = 1 光核 |
| 完美弹反回能 | 12 | 8-10 次完美弹反可攒满一池，奖励精准操作 |
| 普通弹反回能 | 6 | 完美收益的一半，降低新手挫败但保留高手优势 |
| 超频普攻消耗 | 3 | 低成本清怪，一池能量约 33 次命中，鼓励玩家愿意试用 |
| 超频闪避消耗 | 8 | 中等成本保命，一池能量约 12 次，用于危险时止损 |
| 超频弹反消耗 | 15 | 高成本高回报，一池能量约 6 次，强化弹反但不取代普通弹反 |
| 光核兑换比例 | 100 能量 = 1 光核 | 与能量池上限一致，便于把战斗资源转成构筑进度 |
| 光核存储上限 | P0/P1 前期 1 颗 | 迫使玩家在祭坛节点花掉光核，避免早期囤积过多天赋次数 |
| 天赋兑换成本 | 1 颗完整光核 | 与祭坛三选一节奏绑定 |

如果 playtest 中玩家仍然倾向于不用超频，优先加强超频动作效果。继续降低消耗会削弱“生存 vs 成长”的张力。

## 5. 与其他系统的联系

**与玩家行为系统**：玩家行为系统负责执行动作并发出弹反成功、攻击命中、闪避触发、超频动作请求等事件。能量系统只接收事件并结算资源，不接管状态机。→ [[策划案_玩家行为系统]]

**与超频动作**：OverclockSystem 是能量系统的动作强化层，负责判断超频请求是否成立、是否降级为普通动作、何时扣能、触发何种 P0 强化反馈。→ [[DESIGN_能量系统#五、超频动作]]

**与光核与祭坛**：光核是能量的战后成长形态。结晶点负责把当前能量转化为光核进度，祭坛负责消耗完整光核触发天赋三选一。→ [[策划案_光核与祭坛]]

**与天赋系统**：天赋后续会改写能量收益、超频消耗、超频触发条件和光核兑换效率。P0 不做这些改写，但能量系统必须保留清晰接口。→ [[策划案_天赋系统]]

**与 HUD**：HUD 只监听 `energy_changed`、`light_core_changed`、`crystallization_completed` 等结果信号，不直接监听玩家动作改 UI。这样后续能量来源变多时，HUD 不需要重写规则。

## 6. 美术/音效需求

P0 阶段的能量系统反馈以“能判断”为准，不追求正式特效。

| 需求 | P0 最低标准 | 说明 |
|------|-------------|------|
| 能量条 | 简单进度条 + 数字 | 能看出当前能量、增加、消耗、满值 |
| 光核显示 | 数字或小图标 + 进度 | 能看出 0/1、进度变化和已满状态 |
| 超频普攻反馈 | 玩家或攻击判定短暂变色/闪光 | 告诉玩家本次攻击是超频版 |
| 超频闪避反馈 | 残影、变色或更亮的位移轨迹 | 告诉玩家本次闪避付费强化 |
| 超频弹反反馈 | 更强闪光、冲击波或音效占位 | 区分普通弹反和超频弹反 |
| 结晶点状态 | 可互动高亮、已使用状态 | 玩家能判断是否可以结晶 |

音效最低需求：

| 音效 | 用途 |
|------|------|
| 能量增加 | 弹反回能时提示资源变化 |
| 超频动作触发 | 区分普通动作与超频动作 |
| 能量不足降级 | 轻提示即可，避免玩家以为输入失效 |
| 结晶完成 | 提示能量已转化为光核进度 |
| 光核完成 | 提示已获得 1 颗完整光核 |

## 7. 程序需求

技术方案遵循 [[INDEX_开发路线图#一、Godot 设计哲学速查]]：数据集中在 RunData，动作来源由 Player 提供，系统间通过信号解耦，HUD 只监听结果。

### 7.1 模块结构与归属

P0-3 先实现战斗中的能量获取、超频动作和能量 HUD；P0-5 再接入光核结晶、结晶点和光核 HUD。

建议节点与脚本归属：

```text
Autoload
├── RunData
└── EventBus

Player
├── EnergySystem
└── OverclockSystem

HUD
├── EnergyHUD
└── LightCoreHUD（P0-5 接入）

CombatRoom / TestRoom
└── CrystallizationPoint（P0-5 接入）
```

模块职责：

| 模块 | 建议归属 | 职责 | 不负责 |
|------|----------|------|--------|
| RunData | Autoload | 保存能量和光核数据，提供加能量、消耗、结晶、花费光核接口 | 判断动作、播放反馈、更新 UI |
| EnergySystem | Player 子节点 | 监听基础回能事件，将弹反成功转化为能量收入 | 保存长期数据、处理超频强化 |
| OverclockSystem | Player 子节点 | 判断超频动作是否成立、是否降级、何时扣能 | 读取玩家输入、控制玩家状态机 |
| EnergyHUD | UI 节点 | 显示当前能量和能量变化 | 判断能量规则 |
| LightCoreHUD | UI 节点 | 显示光核数量与进度 | 判断结晶规则 |
| CrystallizationPoint | 房间交互物 | 在战斗结束后提供结晶互动入口 | 保存永久资源、直接发天赋 |

设计边界：
- Player 负责动作和输入，不直接修改能量数值。
- EnergySystem / OverclockSystem 负责把动作事件翻译成 RunData 调用。
- RunData 是唯一能改 `energy`、`light_core_progress`、`light_cores` 的地方。
- HUD 只读结果信号，不从玩家动作推断数值。

### 7.2 P0-3 实现流程

[弹反回能流程]

1. Player 的 Parry 状态判定弹反成功。
2. Player 或弹反状态发出 `parry_succeeded(is_perfect)`。
3. EnergySystem 监听该信号。
4. EnergySystem 根据 `is_perfect` 选择 12 或 6。
5. EnergySystem 调用 `RunData.add_energy(amount)`。
6. RunData 夹取能量范围并发出 `energy_changed(current, max_energy)`。
7. EnergyHUD 监听 `energy_changed` 并刷新显示。

[超频动作流程]

1. Player 在动作开始时读取 `overclock` 是否按住。
2. 若按住，Player 向 OverclockSystem 提交 `action_type`。
3. OverclockSystem 根据动作类型和当前能量判断是否允许超频。
4. 若能量不足，动作降级为普通版，并发出 `overclock_action_failed(action_type, "not_enough_energy")`。
5. 若能量足够，动作标记为超频版，但扣能时机由动作类型决定。
6. 超频动作达到扣能条件后，OverclockSystem 调用 `RunData.consume_energy(cost)`。
7. 扣能成功后发出 `overclock_action_triggered(action_type, cost)`，供视觉、音效、HUD 反馈使用。

P0-3 必须先跑通：
- 弹反成功后回能。
- 超频普攻命中后扣能。
- 超频闪避触发时扣能。
- 超频弹反成功后按结算顺序扣能。
- 能量 HUD 正确显示。

### 7.3 P0-5 实现流程

[光核结晶流程]

1. 房间清怪系统发出 `all_enemies_cleared` 或 `room_cleared`。
2. CrystallizationPoint 变为可见或可互动。
3. 玩家进入交互范围后显示交互提示。
4. 玩家按下 `interact`。
5. CrystallizationPoint 检查光核是否已满。
6. 未满时调用 `RunData.crystallize()`。
7. RunData 将当前能量转化为光核进度，并清空当前能量。
8. RunData 发出 `energy_changed` 与 `light_core_changed`。
9. 若完成 1 颗光核，发出 `crystallization_completed`。
10. LightCoreHUD 与 EnergyHUD 分别刷新。

P0-5 必须保留两条分支：
- 玩家互动结晶点：能量转为光核进度。
- 玩家不互动结晶点：能量保留，不发生结晶。

光核已满时：
- CrystallizationPoint 不调用 `RunData.crystallize()`。
- 当前能量不清零。
- P0 可先用简单提示或禁用高亮表达“光核已满”。

### 7.4 RunData 数据

| 数据 | 类型 | 默认值 | 说明 |
|------|------|-------:|------|
| energy | int | 0 | 当前能量 |
| max_energy | int | 100 | 当前能量池上限 |
| light_core_progress | float | 0.0 | 当前光核进度，0.0-1.0 |
| light_cores | int | 0 | 已完成光核数量 |
| max_light_cores | int | 1 | 光核存储上限 |
| perfect_parries | int | 0 | 完美弹反次数统计 |

RunData 接口需求：

| 接口 | 返回 | 说明 |
|------|------|------|
| reset_run() | void | 重置当局能量、光核、统计数据 |
| add_energy(amount) | void | 增加能量，夹到 0 ~ max_energy |
| consume_energy(amount) | bool | 能量足够则扣除并返回 true，不足则返回 false |
| crystallize() | bool | 将当前能量转为光核进度；成功返回 true，不能结晶返回 false |
| spend_light_core() | bool | 花费 1 颗光核，供祭坛兑换天赋 |

接口约束：
- `add_energy()` 和 `consume_energy()` 必须广播 `energy_changed`。
- `crystallize()` 必须先确认光核未满；光核已满时不能清空能量。
- `crystallize()` 成功后必须广播 `energy_changed` 和 `light_core_changed`。
- 每完成 1 颗光核时广播 `crystallization_completed`。

### 7.5 Balance 数据

P0 可以先用 Resource 或集中配置表管理以下数值，避免写死在多个脚本中：

| 字段 | 类型 | 默认值 | 说明 |
|------|------|-------:|------|
| perfect_parry_energy | int | 12 | 完美弹反回能 |
| normal_parry_energy | int | 6 | 普通弹反回能 |
| overclock_attack_cost | int | 3 | 超频普攻消耗 |
| overclock_dodge_cost | int | 8 | 超频闪避消耗 |
| overclock_parry_cost | int | 15 | 超频弹反消耗 |
| light_core_energy_ratio | float | 100.0 | 100 能量 = 1 光核 |

如果 P0 为了速度先沿用 `PlayerStats` 中的弹反能量字段，也可以，但超频消耗和光核比例应尽快集中到 `data/balance/` 下，避免分散。

### 7.6 信号需求

优先复用或新增以下 EventBus 信号：

| 信号 | 参数 | 触发时机 |
|------|------|----------|
| parry_succeeded | is_perfect: bool | 玩家弹反成功，EnergySystem 监听回能 |
| parry_failed | — | 玩家弹反窗口结束但未成功，OverclockSystem 用于清理超频弹反候选 |
| attack_finished | — | 普攻流程结束，用于清理未命中的超频普攻候选 |
| dodge_started | — | 闪避开始，OverclockSystem 用于结算触发即扣的超频闪避 |
| overclock_action_requested | action_type: StringName | 玩家按住超频键开始某个动作 |
| overclock_action_accepted | action_type: StringName, cost: int | 当前能量足够，本次动作被批准为超频候选，但尚未扣能 |
| overclock_action_triggered | action_type: StringName, cost: int | 超频动作成功触发并扣能 |
| overclock_action_failed | action_type: StringName, reason: StringName | 超频请求失败或降级，如能量不足 |
| energy_changed | current: int, max_energy: int | 能量变化 |
| light_core_changed | current: float, max_cores: float | 光核进度或数量变化 |
| crystallization_completed | — | 完成 1 颗光核 |

说明：`overclock_action_requested` 只代表玩家请求，不代表一定扣能。`overclock_action_accepted` 代表“这次动作有资格进入超频版”。真正扣能与强化成功应由 `overclock_action_triggered` 表达。

### 7.7 当前工程迁移点

当前 Godot 工程已有 P0-2 玩家行为基础，但超频入口仍是旧“开关模式”。进入 P0-3 实现能量系统前，需要先迁移以下内容，避免新策划案与代码语义冲突。

现有工程背景：

| 文件 | 当前状态 | 需要调整 |
|------|----------|----------|
| `autoloads/event_bus.gd` | 已有 `overclock_toggled(is_active)` | 新增或替换为 `overclock_action_requested` / `overclock_action_triggered` / `overclock_action_failed` |
| `entities/player/player.gd` | 已有 `is_overclock_active`、`toggle_overclock()`、`set_overclock_active()` | 废弃开关状态，改为动作开始时读取 `overclock` 是否按住，并提交动作类型 |
| `tests/player_overclock_entry_test.gd` | 测试“开启/关闭超频” | 改写为“按住超频键 + 普攻/闪避/弹反请求超频动作”的测试 |
| `data/player/player_stats.gd` | 已有弹反回能、基础动作数值 | 可暂存弹反回能；超频消耗建议迁移到 Balance 数据 |
| `autoloads/run_data.gd` | 已有能量、光核、结晶接口 | 需要确认 `crystallize()` 光核已满时不清空能量，并按本案返回成功/失败 |

迁移原则：
- 不再保存 `is_overclock_active` 作为持续状态。
- 按住 `overclock` 本身不发“开启”事件，不扣能。
- 只有动作开始时才采样超频输入。
- `overclock_action_requested` 表示请求，不表示一定扣能。
- 超频动作成功、失败、降级要有独立结果信号，供反馈和测试使用。
- 旧测试需要改写测试目标，确保测试语义从“开关模式”迁移到“动作修饰输入”。

### 7.8 EnergySystem 需求

EnergySystem 只负责基础能量收入。P0 阶段只监听 `parry_succeeded`：

| 输入事件 | 条件 | 处理 |
|----------|------|------|
| parry_succeeded(true) | 完美弹反 | 调用 `RunData.add_energy(12)` |
| parry_succeeded(false) | 普通弹反 | 调用 `RunData.add_energy(6)` |

实现约束：
- EnergySystem 不保存能量值。
- EnergySystem 不直接更新 HUD。
- EnergySystem 不处理超频动作消耗。
- 后续天赋回能可以扩展到 EnergySystem，但 P0 不实现。

### 7.9 OverclockSystem 需求

OverclockSystem 至少需要处理三类动作：

| action_type | 成立条件 | 扣能时机 | 失败处理 |
|-------------|----------|----------|----------|
| attack | 能量 ≥ 3，且本次普攻命中敌人 | 命中后 | 空挥不扣能；能量不足则普通攻击 |
| dodge | 能量 ≥ 8，且闪避开始 | 闪避触发时 | 能量不足则普通闪避 |
| parry | 能量 ≥ 15，且弹反成功 | 弹反成功后 | 弹反失败不扣能；能量不足则普通弹反 |

动作成功强化后，OverclockSystem 调用 RunData `consume_energy(cost)`，并发出超频动作结果信号，供视觉与音效系统播放反馈。

实现约束：
- `overclock_action_requested` 只表示玩家提出超频请求，实际扣能由后续结果信号表达。
- 能量不足时必须降级为普通动作，不阻止玩家动作本身执行。
- 超频普攻空挥不扣能，并在攻击结束时清理本次超频候选。
- 超频弹反失败不扣能。
- 超频闪避触发即扣能。
- 超频弹反采用“基础回能先结算，超频消耗后结算”；实现上不要依赖信号监听顺序，可延后一拍再扣能。
- OverclockSystem 不直接播放特效，只发结果信号；视觉系统或 Player 表现层监听结果。

### 7.10 光核结晶交互

CrystallizationPoint 需要：

- 在房间清空后可见或可互动。
- 玩家进入范围后显示交互提示。
- 按下 `interact` 后调用 RunData 结晶接口。
- 结晶成功后播放反馈并进入已使用状态。
- 光核已满时不清空能量。

光核满时禁止结晶是 P0 必须规则。否则玩家可能误操作，把能量清零却没有得到任何构筑进度。

### 7.11 HUD 需求

EnergyHUD：
- 监听 `energy_changed`。
- 显示当前能量和最大能量。
- 能量增加、消耗时有简单变化反馈。

LightCoreHUD：
- P0-5 接入。
- 监听 `light_core_changed` 与 `crystallization_completed`。
- 显示当前光核数量和进度。

HUD 不判断弹反、超频或结晶规则。

## 8. 验收标准

- [ ] RunData 能保存 `energy`、`max_energy`、`light_core_progress`、`light_cores`、`max_light_cores`。
- [ ] 完美弹反成功后能量增加 12，并刷新 HUD。
- [ ] 普通弹反成功后能量增加 6，并刷新 HUD。
- [ ] 当前能量不会超过 `max_energy`，也不会低于 0。
- [ ] 按住超频键普攻，空挥不扣能。
- [ ] 按住超频键普攻并命中敌人后，消耗 3 能量并触发超频普攻反馈。
- [ ] 按住超频键闪避时，立即消耗 8 能量并触发超频闪避反馈。
- [ ] 按住超频键弹反失败时，不消耗能量。
- [ ] 按住超频键弹反成功时，先获得基础弹反能量，再消耗 15 能量。
- [ ] 能量不足时，超频动作降级为普通动作，不扣能，不播放超频成功反馈。
- [ ] 战斗结束后结晶点出现或变为可互动。
- [ ] 与结晶点互动时，当前能量按比例转化为光核进度，能量清零。
- [ ] 不与结晶点互动时，当前能量保留。
- [ ] 光核满时，结晶点不会清空当前能量。
- [ ] 光核进度达到 1.0 时，增加 1 颗光核并触发完成反馈。
- [ ] HUD 只通过能量与光核结果信号刷新，不直接从玩家动作推断数值。

## 9. 待验证问题

1. 超频普攻消耗 3 是否足够便宜，能否鼓励玩家用它清怪。
2. 超频闪避消耗 8 是否会让玩家愿意用它救命。
3. 超频弹反净亏 3 / 9 是否符合“高回报但有代价”的预期。
4. 玩家是否仍倾向于囤能量不用超频；如果是，优先加强超频效果。
5. 能量不足降级为普通动作时，玩家是否能理解系统反馈来自“能量不足”。
6. 结晶点“不互动就保留能量”的规则是否足够直观。
7. 光核满时阻止结晶是否需要明确 UI 提示。
8. 能量 HUD 是否足以让玩家理解“战斗资源”和“构筑进度”的关系。
