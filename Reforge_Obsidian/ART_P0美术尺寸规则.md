## 一、定位

本文记录 Reforge P0 阶段的美术尺寸规则，用于玩家、敌人、Boss、交互物、地图和背景资产的生产与 Godot 接入。

核心原则：

```text
先定游戏内显示大小，再定高清源图画布，最后在 Godot 缩放到显示大小。
```

资产生产不按生成图原始尺寸决定游戏内大小。Godot 接入时根据目标显示尺寸反算 `scale`。

---

## 二、基础画面规则

| 项目 | P0 基准 |
|------|---------|
| 逻辑视口 | 1280 × 720 |
| 屏幕比例 | 核心构图按 16:9 |
| 全屏适配 | `expand`，非 16:9 屏幕显示更多边缘内容 |
| 逻辑 tile | 32 × 32 px |
| 标准房间 | 64 × 36 tile，约 2048 × 1152 px |
| 摄像机 | 决定能看到多少战场，不用于修正角色大小 |

`Camera Zoom` 用来调整战场可见范围。角色、敌人、Boss 和交互物的大小优先通过各自的显示尺寸与 `scale` 控制。

---

## 三、角色与怪物规则

| 类型 | 源图画布 | 源图主体尺寸 | 游戏内显示目标 |
|------|----------|--------------|----------------|
| 玩家 | 512 × 512 透明画布 | 整体约 320-380 px 高 | 约 135-145 px 高 |
| 普通敌人 | 512 × 512 透明画布 | 约 240-320 px 宽 / 高 | 约 80-130 px |
| 锈犬 | 512 × 512 透明画布 | 主体约 260-320 px 宽 | 约 115-120 px 宽 |
| 精英敌人 | 768 × 768 透明画布 | 约 420-560 px | 约 160-220 px |
| Boss | 1024 × 1024 或更大透明画布 | 按 Boss 类型单独定 | 约 260-420 px |

当前 P0 校准基准：

| 对象 | 当前源图 | 当前缩放 | 当前游戏内显示 |
|------|----------|----------|----------------|
| 玩家 | 512 × 512 px，主体约 358 px 高 | `0.394` | 主体约 141 px 高 |
| 锈犬 | 512 × 512 px，主体约 300 px 宽 | `0.390` | 主体约 117 px 宽 |

当前 512 × 512 源图是按 P0 尺寸规则生成的测试资产，可作为后续动画生产的画布和显示目标参考。旧临时裁切图继续保留，用于对比统一画布前后的清晰度和显示大小。

当前新旧资产对比：

![[附件/p0_canvas512_player_hound_compare_v2.png]]

这张图说明：统一 512 × 512 画布后，角色在源图里拥有更高清的主体尺寸；接入 Godot 时仍然缩放到原本确认过的游戏内显示目标。尺寸规则追求的是“显示大小稳定 + 源图清晰 + 锚点一致”，不是让生成图原始尺寸直接决定游戏内大小。

## 四、交互物规则

| 类型 | 源图画布 | 游戏内显示目标 |
|------|----------|----------------|
| 结晶点 | 256 × 256 透明画布 | 约 64-96 px |
| 祭坛 / 锻造台 | 512 × 512 透明画布 | 约 140-220 px |
| 出口门 / 房间门 | 512 × 512 或 768 × 768 透明画布 | 约 180-300 px |
| 小掉落物 / 光核碎片 | 128 × 128 或 256 × 256 透明画布 | 约 24-48 px |
| UI 图标 | 128 × 128 或 256 × 256 透明画布 | UI 内按需求缩放 |

交互物需要保留脚底、底座或落点信息，便于判断碰撞位置和可交互范围。

---

## 五、地图与背景规则

| 类型 | 规则 |
|------|------|
| 地板 tile | 32 × 32 px 逻辑 tile |
| 地面大块材质 | 可用 256 × 256 / 512 × 512 纹理拼接 |
| 墙体 / 边缘 | 按 32px tile 对齐，允许绘制超过 tile 的高度 |
| 背景装饰 | 不参与碰撞时可自由尺寸，但要服务 32px 网格 |
| 房间边缘 | `expand` 下会多显示边缘内容，边缘不能空 |

地图资产可以使用 tile、拼接纹理和大型装饰混合制作。参与碰撞和行走判断的部分必须服从 32px 逻辑网格。

## 六、Godot 缩放规则

同一个角色只允许一个显示缩放基准：

```text
玩家：同一套动作共用一个 visual_scale
锈犬：同一套动作共用一个 enemy_visual_scale
Boss：每个 Boss 单独定一个 boss_visual_scale
交互物：按类型定显示尺寸
```

接入资产时按公式计算：

```text
scale = 目标游戏内显示尺寸 / 源图主体尺寸
```

例：

```text
玩家目标显示高度 140px，源图主体高度 350px
scale = 140 / 350 = 0.4

锈犬目标显示宽度 118px，源图主体宽度 300px
scale = 118 / 300 ≈ 0.39
```

同一个角色的所有动作帧必须满足：

```text
同一画布尺寸
同一角色大小
同一脚底锚点
同一朝向规则
同一 Godot 缩放
```

## 七、生图与动画规则

生图和动画分帧按源图画布制作，不按最终游戏内显示尺寸制作。

| 对象 | 生图 / 分帧要求 |
|------|----------------|
| 玩家 | 512 × 512 透明画布，角色整体保持 320-380 px 高 |
| 锈犬 | 512 × 512 透明画布，主体保持 260-320 px 宽 |
| 结晶点 | 256 × 256 透明画布 |
| 祭坛 | 512 × 512 透明画布 |
| Boss | 1024 × 1024 或更大透明画布 |

Prompt 中需要明确：

```text
consistent character size across all frames,
centered on a transparent 512x512 canvas,
stable foot anchor point,
same camera angle and proportions
```

中文含义：

```text
所有帧保持同一角色大小；
放在统一透明画布里；
脚底锚点稳定；
视角和比例一致。
```

---

## 八、当前待验证

1. 玩家和锈犬按当前显示目标进入即梦动画流程后，动作帧是否能保持稳定体型。
2. 512 × 512 统一画布是否足够容纳玩家攻击、闪避和弹反的大动作。
3. 锈犬扑击动作是否需要横向更大的画布，或用 768 × 512 动作画布单独处理。
4. `expand` 全屏适配下，房间边缘、UI 和刷新点是否需要额外安全区规范。
