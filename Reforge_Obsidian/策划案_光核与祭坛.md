## 1. 设计目的

光核与祭坛系统负责把战斗中的资源表现转化为本局构筑机会。玩家在战斗中通过弹反和后续构筑获得能量，也会通过超频动作消耗能量。战斗结束后，剩余能量可以通过结晶点转化为光核进度；完整光核可以在祭坛兑换天赋选择。

这个系统让战斗表现进入构筑节奏：

```text
战斗剩余能量 → 结晶为光核进度 → 完整光核 → 祭坛兑换 → 天赋选择
```

P0-5 的目标是验证两个空间行为：

- 玩家走向结晶点，主动把当前能量转化为光核进度。
- 玩家走向祭坛，消耗完整光核请求天赋选择。

结晶点和祭坛都放在场景中，以交互物承载资源决策。玩家通过移动和互动完成选择，战后流程保持连续。

## 2. 系统定位

光核与祭坛位于能量系统和天赋系统之间。

| 模块 | P0-5 职责 | 后续扩展 |
|------|-----------|----------|
| RunData 光核数据 | 保存光核进度、完整光核、光核上限 | Meta 提升上限、跨局统计 |
| 结晶点 | 清房后互动，将当前能量转化为光核进度 | 不同结晶效率、事件房改写 |
| LightCoreHUD | 显示光核数量、进度和已满状态 | 完整 UI 美术、阶段命名切换 |
| 祭坛 | 消耗 1 光核，发出天赋选择请求 | 祭坛变体、折扣、特殊代价 |
| 天赋入口 | P0-5 只发请求或打开占位入口 | P0-6 实装三选一与天赋效果 |

P0-5 不实现完整天赋池。祭坛在 P0-5 只完成“消耗光核并请求天赋选择”的接口，真正的天赋三选一、天赋数据和效果系统归 [[策划案_天赋系统]]。

## 3. 具体规则

### 3.1 结晶点

[结晶点出现]

触发条件：战斗房间清空，房间系统发出 `all_enemies_cleared` 或 `room_cleared`。

序列：
1. 结晶点从不可互动状态变为可互动状态。
2. 结晶点显示 P0 占位外观或高亮。
3. 玩家进入交互范围后显示交互提示。
4. 玩家按下 `interact` 后，请求结晶。

结束后：等待玩家互动；若玩家直接离开房间，当前能量保留。

设计理由：结晶点是战斗房间的结算入口。它跟随清房节奏出现，让玩家在安全状态下处理资源。

[光核结晶]

触发条件：玩家在结晶点交互范围内按下 `interact`，且结晶点处于可互动状态。

序列：
1. 检查当前能量是否大于 0。
2. 检查完整光核是否低于存储上限。
3. 调用 `RunData.crystallize()`。
4. 将当前能量按 `当前能量 / 最大能量` 转化为光核进度。
5. 当前能量清零。
6. 光核进度达到 1.0 时，增加 1 颗完整光核。
7. 光核达到存储上限时，额外进度不继续累积。
8. 广播 `energy_changed`、`light_core_changed`，必要时广播 `crystallization_completed`。
9. 结晶点进入已使用状态。

结束后：EnergyHUD 和 LightCoreHUD 刷新。

规则：
- 当前能量为 0 时，结晶失败，当前能量保持 0。
- 光核已满时，结晶失败，当前能量保持不变。
- 结晶点每次清房只允许成功结晶一次。
- 玩家不互动结晶点时，当前能量保留。

设计理由：结晶是主动成长选择。成功结晶会清空战斗能量，换取光核进度；跳过结晶会保留后续战斗预算。

### 3.2 光核

[光核进度]

触发条件：`RunData.crystallize()` 成功。

序列：
1. `converted_progress = energy / max_energy`。
2. `light_core_progress += converted_progress`。
3. 若 `light_core_progress >= 1.0`，增加完整光核。
4. 完整光核增加后，扣除对应进度。
5. 完整光核达到上限时，光核进度归零或夹到可显示范围。

结束后：广播 `light_core_changed(current, max_cores)`。

P0 显示方式：
- `current = light_cores + light_core_progress`。
- `max_cores = max_light_cores`。
- 前期上限为 1。

设计理由：光核进度允许碎片化积累。玩家不需要单场战斗攒满 100 能量，也能逐步推进构筑。

### 3.3 祭坛

[祭坛交互]

触发条件：玩家在祭坛交互范围内按下 `interact`。

序列：
1. 检查 `RunData.light_cores >= 1`。
2. 若光核不足，交互失败，提示缺少完整光核。
3. 若光核足够，调用 `RunData.spend_light_core()`。
4. 成功花费 1 颗完整光核后，发出天赋选择请求。
5. P0-5 若天赋系统尚未完成，可打开占位提示或只发信号。
6. P0-6 接入后，由天赋系统监听请求并打开三选一 UI。

结束后：LightCoreHUD 刷新；天赋选择流程进入 P0-6 负责的界面。

设计理由：祭坛把完整光核转化为构筑选择。P0-5 先保证资源消费和系统接口成立，天赋生成逻辑在后续阶段接入。

### 3.4 交互优先级

当玩家同时处于多个交互物范围时，P0 可采用距离最近优先。

交互优先级：

[3-高] 祭坛
- 玩家拥有完整光核时，优先处理祭坛兑换。
- 玩家没有完整光核时，提示光核不足。

[2-中] 结晶点
- 玩家有能量且光核未满时，处理结晶。
- 能量为 0 或光核已满时，提示失败原因。

[1-低] 出口门
- 由测试房间或房间推进系统处理。

设计理由：测试房间可能同时放置结晶点和祭坛。明确交互优先级可以减少同键交互冲突。

## 4. 数值与依据

| 项目 | P0 数值 | 依据 |
|------|--------:|------|
| 能量池上限 | 100 | 与光核兑换比例一致，便于理解 |
| 光核兑换比例 | 100 能量 = 1 光核 | 线性转换，10 能量 = 0.1 光核 |
| 光核存储上限 | 1 | 前期控制天赋兑换节奏 |
| 祭坛兑换成本 | 1 完整光核 | 每次兑换对应一次明确构筑机会 |
| 结晶点交互半径 | 约 48px | P0 可读、易触发，避免过小 |
| 祭坛交互半径 | 约 56px | 祭坛体积更大，交互范围略宽 |

调参原则：
- 玩家频繁满光核却找不到祭坛时，优先调整祭坛分布，不先提高光核上限。
- 玩家长期无法获得完整光核时，优先检查敌人能量机会和超频消耗，不先降低祭坛成本。
- 玩家误触结晶导致战斗能量清空时，优先加强提示和确认反馈，不先取消主动结晶规则。

## 5. 与其他系统的联系

**与能量系统**：能量系统提供可结晶的当前能量。结晶点只调用 RunData 接口，不直接计算弹反回能或超频消耗。→ [[策划案_能量系统]]

**与天赋系统**：祭坛发出天赋选择请求。天赋池、三选一 UI、天赋效果与权重规则归天赋系统实现。→ [[策划案_天赋系统]]

**与房间系统**：结晶点在房间清空后启用。正式关卡中，祭坛是非战斗节点；P0 测试房间可同时摆放结晶点和祭坛，用于验证完整闭环。

**与 HUD**：EnergyHUD 显示当前能量，LightCoreHUD 显示光核进度和完整光核。HUD 只监听结果信号，不判断结晶或祭坛规则。

**与世界观命名**：P0 默认使用前期主角视角命名。后续叙事阶段切换时，同一机制切到后期真相命名。→ [[DESIGN_世界观设计文档#3.2 双层命名切换系统]]

P0-5 以可读状态为目标。正式粒子、混音和高级演出归 P0-7 / P2。

### 6.3 音效与反馈

P0-5 必须保留反馈事件触发点。音频播放可在 P0-7 统一接入，也可在 P0-5 用占位 AudioStreamPlayer 临时验证交互读感，最终仍应收敛到反馈系统。

| 反馈 | 触发点 | P0-5 要求 |
|------|--------|----------|
| 结晶成功 | `crystallization_completed` 或结晶成功返回 | 有视觉变化或提示 |
| 光核进度变化 | `light_core_changed` | HUD 刷新 |
| 光核已满 | 结晶失败 reason = `light_core_full` | 不清空能量并提示 |
| 祭坛兑换成功 | `talent_selection_requested` | 发请求或打开占位 |
| 光核不足 | 祭坛交互失败 reason = `not_enough_light_core` | 提示失败原因 |

## 7. 程序需求

技术方案遵循 [[INDEX_开发路线图#一、Godot 设计哲学速查]]：数据集中在 RunData，交互物负责触发请求，系统间通过信号解耦，HUD 只监听结果。

### 7.1 场景结构

建议节点：

```text
Interactables
├── CrystallizationPoint (Area2D / Node2D)
│   ├── Sprite2D
│   ├── InteractionArea
│   └── PromptAnchor
└── Altar (Area2D / Node2D)
    ├── Sprite2D
    ├── InteractionArea
    └── PromptAnchor

HUD
└── LightCoreHUD
```

P0 可先把结晶点和祭坛直接放在测试房间中。正式房间系统接入后，结晶点由清房事件启用，祭坛由路线节点生成。

### 7.2 RunData 需求

已有字段：

| 字段 | 类型 | P0 值 | 说明 |
|------|------|------:|------|
| `light_core_progress` | float | 0.0 | 0.0-1.0 的当前进度 |
| `light_cores` | int | 0 | 完整光核数量 |
| `max_light_cores` | int | 1 | P0 光核上限 |

接口需求：

| 接口 | 返回 | 说明 |
|------|------|------|
| `crystallize()` | bool | 成功结晶返回 true；能量为 0 或光核已满返回 false |
| `spend_light_core()` | bool | 成功花费 1 光核返回 true；光核不足返回 false |

接口约束：
- `crystallize()` 成功时清空当前能量。
- `crystallize()` 失败时不修改当前能量。
- 光核已满时，`crystallize()` 必须返回 false。
- `crystallize()` 成功后广播 `energy_changed` 与 `light_core_changed`。
- 每完成 1 颗完整光核时广播 `crystallization_completed`。
- `spend_light_core()` 成功后广播 `light_core_changed`。

### 7.3 当前工程迁移点

当前工程已有 RunData 光核字段和基础接口，但 P0-5 实现前需要修正：

| 文件 | 当前状态 | 需要调整 |
|------|----------|----------|
| `autoloads/run_data.gd` | `crystallize()` 当前返回 void | 改为 bool，供结晶点判断成功 / 失败 |
| `autoloads/run_data.gd` | 光核已满时仍可能清空当前能量 | 光核已满时直接返回 false，保留能量 |
| `autoloads/event_bus.gd` | 已有 `light_core_changed`、`crystallization_completed` | 可补交互失败 / 祭坛请求信号 |
| `localization/translations.csv` | 已有核心双层命名 key | 需补失败提示 key |

### 7.4 EventBus 信号需求

已有信号：

| 信号 | 参数 | 用途 |
|------|------|------|
| `light_core_changed` | current: float, max_cores: float | 光核数量或进度变化 |
| `crystallization_completed` | — | 完成 1 颗完整光核 |
| `energy_changed` | current: int, max_energy: int | 结晶成功后刷新能量 |

建议新增：

| 信号 | 参数 | 用途 |
|------|------|------|
| `crystallization_failed` | reason: StringName | 能量为 0、光核已满等失败提示 |
| `altar_activated` | — | 祭坛成功消耗光核 |
| `altar_failed` | reason: StringName | 光核不足等失败提示 |
| `talent_selection_requested` | — | 交给 P0-6 天赋系统打开三选一 |

P0-5 的交互物只发事件，不直接生成天赋结果。

### 7.5 CrystallizationPoint 需求

状态：

| 状态 | 行为 |
|------|------|
| Disabled | 不显示或不可互动 |
| Available | 清房后可互动 |
| Used | 成功结晶后不可再次互动 |
| Blocked | 光核已满或能量不足时保持可读提示 |

规则：
- 监听 `all_enemies_cleared` 或由房间系统直接调用启用。
- 玩家进入交互范围后显示提示。
- 玩家按 `interact` 时调用 `RunData.crystallize()`。
- 成功后进入 Used。
- 失败时保持 Available 或 Blocked，并发出失败提示。

### 7.6 Altar 需求

状态：

| 状态 | 行为 |
|------|------|
| Available | 可检测玩家互动 |
| WaitingTalentSystem | 已成功消耗光核，等待天赋系统响应 |
| Used | 本次祭坛兑换完成 |
| Blocked | 光核不足 |

规则：
- 玩家进入交互范围后显示提示。
- 玩家按 `interact` 时调用 `RunData.spend_light_core()`。
- 成功后发出 `talent_selection_requested`。
- P0-6 未完成时，显示占位提示并进入 Used 或保留 Available，具体以测试需要决定。
- 光核不足时不改变光核数据。

### 7.7 LightCoreHUD 需求

LightCoreHUD：
- 监听 `light_core_changed`。
- 显示完整光核数量和进度。
- P0 可显示为 `当前 / 上限` 文本，例如 `0.35 / 1`。
- 完整光核达到上限时显示已满状态。
- 不直接读取玩家动作，不判断结晶规则。

### 7.8 测试需求

P0-5 至少需要以下测试：

- `RunData.crystallize()` 在 50 能量时生成 0.5 光核进度并清空能量。
- `RunData.crystallize()` 在 100 能量时生成 1 完整光核并触发 `crystallization_completed`。
- 光核已满时调用 `crystallize()` 返回 false，能量不清空。
- `spend_light_core()` 在有光核时返回 true 并减少 1。
- `spend_light_core()` 在无光核时返回 false。
- LightCoreHUD 只通过 `light_core_changed` 刷新。
- 结晶点成功后进入 Used，不能重复结晶。
- 祭坛成功后发出天赋选择请求。

## 8. 验收标准

- [ ] RunData 的 `crystallize()` 返回 bool，成功 / 失败分支可被交互物判断。
- [ ] 当前能量大于 0 且光核未满时，结晶点互动会把能量转为光核进度并清空能量。
- [ ] 玩家不互动结晶点时，当前能量保持不变。
- [ ] 当前能量为 0 时，结晶点不会改变光核数据。
- [ ] 光核已满时，结晶点不会清空当前能量。
- [ ] 光核进度达到 1.0 时，完整光核 +1，并触发 `crystallization_completed`。
- [ ] LightCoreHUD 能显示光核数量、进度和已满状态。
- [ ] 祭坛在拥有 1 完整光核时可消耗光核，并发出 `talent_selection_requested`。
- [ ] 祭坛在光核不足时不消耗资源，并给出失败提示。
- [ ] P0-5 不实现天赋池生成；天赋选择请求可由占位 UI 或 P0-6 天赋系统承接。
- [ ] 结晶点与祭坛使用多语言 key，不写死玩家可见文本。
- [ ] 结晶成功、失败、祭坛成功、祭坛失败都有事件触发点，供 P0-7 反馈系统监听。
