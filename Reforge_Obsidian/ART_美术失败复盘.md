## 一、为什么写这篇

这篇记录 Reforge P0 美术推进中的失败尝试和心得。目的不是追责，而是把“为什么做不下去”说清楚，避免后续继续用错误方法消耗时间。

Reforge 是一个 2D 俯视动作 Roguelike，P0 需要展示玩家攻击、格挡、弹反、受伤、锈犬蓄力、扑击、被弹反等状态。美术问题并不是“缺一张好看的图”，而是缺一套能稳定表达状态机的资产生产方法。

---

## 二、尝试过的路线

### 2.1 AI 单帧生图

最初用 AI 生成玩家和锈犬的单帧基准图。结果是单帧观感可以接受，玩家和锈犬的风格也能初步统一。

失败点在于：单帧不能解决动作状态机。攻击、待机、跑步、格挡、受伤需要同一角色在不同动作下保持比例、装备、视角和脚底锚点一致。通用生图工具每次都会重新理解角色，导致头盔、肩甲、武器比例和身体结构变化。

心得：AI 单帧适合确定风格方向，不适合直接承担动作帧生产。

### 2.2 AI 生视频再抽帧

尝试用输入图生成待机、攻击、跑步等动作视频，再从视频中抽关键帧做精灵。实际问题是：视频生成会改变角色细节，动作帧之间的一致性无法稳定控制。

当时设想的完整工作流是：

```text
GPT / Gemini 生角色基准图
→ 即梦 / 可灵根据首帧或首尾帧生动作视频
→ Blender 导入视频并拆成分帧
→ 人工挑选关键帧
→ PS 修图、缩尺寸、统一透明背景
→ Aseprite 合成精灵图
→ Godot 接入 AnimatedSprite2D / AnimationPlayer
```

这条流程的问题不在于步骤不可执行，而在于源头视频不稳定。只要视频阶段出现角色漂移、装备变形、武器比例变化，后面的 Blender、PS、Aseprite 都会变成补救成本，而不是稳定生产。

对 Reforge 来说，攻击、格挡、弹反这些动作必须被玩家读懂，不能只看起来“动了”。如果动作过程中角色造型漂移，反而会破坏状态机可读性。

心得：视频生成适合概念展示，不适合直接作为 P0 状态机动画资产。

### 2.3 AI 精灵图生成工具

也尝试过使用直接生成 spritesheet / 精灵图的 AI 工具，希望绕过“视频抽帧”这一步，直接得到可接入 Godot 的动作序列。

失败点仍然是一致性。即使工具输出形式接近游戏资源，角色在不同帧、不同动作之间仍可能出现比例变化、装备变化、视角漂移或动作不可读的问题。对静态展示来说这些问题可以忍受，但对动作状态机来说会直接影响玩家判断。

心得：AI 精灵图工具可以继续作为候选，但不能默认视为稳定生产管线。它必须先通过“一个角色、一个方向、三个动作”的小样测试。

### 2.4 SVG / 矢量代码生成

尝试参考浏览器游戏中用 SVG / Canvas 生成角色和特效的思路，将玩家改成矢量剪影。这个方向理论上可以解决缩放、改色和部件复用问题。

失败点在于：没有先做出一张审美合格的矢量方案图，就直接在 Godot 里用 `Polygon2D` 拼角色和特效。结果变成廉价占位，既不符合 Reforge 的角色气质，也不能作为作品集展示。

心得：矢量路线的前置条件是先有合格视觉设计。技术能画出形状，不代表能做出好看的角色。

### 2.5 授权素材包

曾考虑用现成 top-down 资源包替换玩家、敌人和地图，以快速跑通 P0 闭环。

这个方案的局限很明显：Reforge 的玩家、锈犬、异力、锈蚀犬甲、结晶点和祭坛都带有世界观特征。素材包很难找到“锈犬”这种专属敌人。强行使用素材包会让作品集展示和项目设计脱节。

心得：素材包可以做临时技术验证，但不能作为 Reforge P0 作品集主视觉方案。

### 2.6 3D 转 2D

3D 模型渲染成 2D 精灵理论上可以解决方向和一致性问题。同一个模型可以渲染多方向、多动作，不会出现 AI 逐帧漂移。

问题在于模型和动画本身仍然需要美术能力。没有合格模型、骨骼和动作，渲染出来也不会好看。AI 可以辅助技术流程，但不能保证独立产出作品集级角色和锈犬模型。

心得：3D 转 2D 是可行管线，但不是低成本捷径。它需要模型来源、绑定、动作和渲染规范。

## 三、核心问题

这轮失败暴露出的核心问题是：P0 需要的是“状态机可视化资产”，不是单独的美术图。

玩家至少要能读出：

- 待机。
- 移动。
- 攻击起手、攻击生效、攻击收束。
- 格挡 / 弹反窗口。
- 弹反成功。
- 受伤。
- 死亡。

锈犬至少要能读出：

- 待机 / 追击。
- 蓄力。
- 扑击。
- 被弹反硬直。
- 受伤。
- 死亡。

如果生产方法不能稳定覆盖这些状态，就算单张图好看，也无法支撑动作游戏 P0 展示。

---

## 四、关键心得

### 4.1 风格图不等于动作资产

一张好看的玩家图只能证明角色方向成立，不能证明动作游戏成立。动作游戏的最低要求是状态变化可读，尤其是攻击、格挡、弹反和受伤。

### 4.2 一致性必须来自资产结构

依靠 prompt 让 AI 每次生成同一个角色并不稳定。一致性应该来自固定资产结构，例如统一的精灵图规格、同一套部件、同一个 3D 模型，或同一套人工修图流程。

### 4.3 P0 不应该伪装成正式美术

如果当前美术能力只能做到原型表现，就应该明确标注为 Prototype Visual。强行把廉价占位包装成正式美术，会降低作品集可信度。

### 4.4 先验证状态机读感，再谈正式美术

P0 当前最需要的是让玩家一眼看懂动作状态。正式风格、八方向、美术精修都应该服从这个目标。

### 4.5 工具链选择要先小样验证

任何新路线都必须先做小样：

```text
一个角色
一个方向
三个动作：idle / attack / parry
```

小样没有达到可接受标准前，不进入主工程。

## 五、后续原则

### 5.1 P0 展示目标

P0 展示目标应调整为：

```text
完整闭环 + 状态机可读 + 明确原型视觉
```

不要把“正式美术完成”作为 P0 成功条件。

### 5.2 主工程只接入已验证资产

主工程不再直接接入未经审美确认的 SVG、多边形、AI 动画或临时图。新美术路线必须先在独立小样中验证，再进入 Godot。

### 5.3 美术探索与 P0 闭环分轨

美术探索和 P0 闭环需要分开推进：

- P0 闭环轨：用清楚的原型视觉展示系统、反馈和流程。
- 美术探索轨：单独验证玩家、锈犬和地图的正式资产生产方法。

### 5.4 状态机表现优先级

当前优先级不是“好看”，而是：

1. 能看出攻击。
2. 能看出格挡 / 弹反。
3. 能看出锈犬蓄力 / 扑击。
4. 能看出受伤 / 死亡。
5. 能看出结晶 / 祭坛 / 天赋确认。

这些成立后，再提升美术风格。

---

## 六、当前结论

最近的美术失败不是单个工具失败，而是目标拆分错误：把“正式视觉风格”“动作状态机资产”“P0 闭环展示”混在一起推进，导致每条路线都承担了超出它能力边界的任务。

后续应先完成状态机可读的 P0 原型，再把正式美术作为独立专项验证。这样既能保护作品集的系统设计价值，也能避免美术生产继续阻塞完整流程。
