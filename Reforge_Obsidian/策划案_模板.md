# 策划案模板

本模板定义 Reforge 项目中所有策划案的写作格式。
结构参考国内游戏行业系统策划案标准流程。
有设计内容的（怪物、系统、Boss 等）使用完整结构。
纯技术任务的（项目基建等）跳过设计部分，从技术需求开始。
行为描述格式参考"顺序描述 + 中断优先级"方法论。

---

## 完整结构

```markdown
# 策划案_XXX

- 所属系统 → [[DESIGN_核心系统设计文档#相关章节]]
- 世界观 → [[DESIGN_世界观设计文档#相关章节]]
- 路线图位置 → [[INDEX_开发路线图#对应章节]]
- 能关联的就关联，没有的不硬凑

## 1. 设计目的

这个东西为什么需要存在，它解决什么设计问题。
包含：为什么这样设计、不这样设计会导致什么问题、参考了什么。
回答"没有它会怎样"比描述"它做什么"更重要。

**示例（锈犬）**：
玩家遇到的第一个敌人，承担弹反教学职责。让从未接触过弹反的玩家在 1-2 次交手后
理解"看前摇 → 按弹反 → 获得奖励"的循环。前摇极其明确、弹反窗口宽松（16帧），
因为教学怪的核心职责是"让玩家成功"而不是"考验玩家"。
如果第一个怪弹反窗口太紧，新手前几次全部失败，会认为弹反太难而放弃使用，
导致能量系统跑不起来，整个游戏核心循环断裂。
对标只狼中前期小兵的劈砍攻击——前摇清晰、节奏稳定，让玩家在低压力下建立肌肉记忆。

## 2. 系统定位

它在整个游戏框架中的位置，和同类设计的区分。

**示例（锈犬）**：
近战追击型怪物的基准线。后续怪物通过和它的对比建立定位——
腐鹿"比锈犬慢但痛"、甲鼠"比锈犬弱但快且多"、巡墙甲"比锈犬硬但慢"。
前期（1-10 房间）单只出现用于教学；中期成群出现配合其他怪制造组合压力；
后期 Build 成型后变成割草对象。

## 3. 具体规则

用行为序列 + 中断优先级格式精确描述行为规则。
系统类策划案（能量系统、天赋系统等）不一定用行为序列格式，
可以用流程描述或规则列表，按内容性质灵活调整。

### 行为序列格式：

[动作名称]
触发条件：
序列：
  1. 阶段名（时间/帧数）→ 具体行为
  2. ...
结束后：→ 回到什么状态

**示例（锈犬·铁甲扑击）**：
[铁甲扑击]
触发条件：进入攻击范围（50px）
序列：
  1. 后蹲蓄力（0.4秒）→ 播蓄力动画，锁定攻击方向
  2. 扑击（0.15秒）→ 向锁定方向位移 80px，生成前方扇形 Hitbox，伤害 8，弹反窗口 16 帧
  3. 落地（0.2秒）→ Hitbox 消失
结束后：→ 回到攻击间隔（2秒）

### 中断优先级表格式：

中断优先级（从高到低）：
  [等级] 状态/事件 → 处理方式

被打断后只有三种去向，必须写明：
- 取消当前动作，进入新状态
- 当前动作暂停，新状态结束后恢复
- 当前动作继续，新效果叠加

**示例（锈犬）**：
[最高] 死亡（HP ≤ 0）→ 取消一切行为 → 播死亡动画
[高] 被弹反 → 取消攻击 → 弹开后退 60px → 硬直 1 秒 → 回到追击
[中] 被击中 → 不打断当前行为（霸体）→ 播闪白 → 扣血
[低] 玩家离开检测范围 → 回到巡逻

### 中断交互格式（两个行为有复杂交互时使用）：

[中断交互] A 状态中触发 B
条件：
处理：
设计理由：

**示例（锈犬·成群交错攻击）**：
[中断交互] 多只锈犬同时存在
条件：3-5 只锈犬各自独立执行行为序列
处理：每只有独立攻击间隔计时器，不做同步，自然错开
设计理由：不刻意编排交错节奏，让独立 AI 自然产生混乱。
玩家觉得纯弹反应对不了时，自然转向闪避，动作切换意识建立。

## 4. 数值与依据

不只列数字，关键数值旁边写为什么。

| 属性 | 数值 | 依据 |
|------|------|------|
| xxx | xx | 为什么是这个数字 |

**示例（锈犬）**：
| 属性 | 数值 | 依据 |
|------|------|------|
| 生命值 | 40 | 玩家攻击力 10 → 4 刀击杀，不会太快没机会弹反，也不会太肉拖节奏 |
| 攻击力 | 8 | 玩家 100HP → 约 12 刀致死，容错空间大，新手有时间学弹反 |
| 弹反窗口 | 16 帧 | 约 0.27 秒，接近反应时间上限，教学怪标准 |
| 硬直时间 | 1 秒 | 弹反后可打 2 刀（攻速 2 次/秒），两次弹反循环可击杀 |

## 5. 与其他系统的联系

和能量系统、天赋系统、关卡节奏、同类设计的关系。

**示例（锈犬）**：
**与能量系统**：前期最稳定的能量来源。每 2 秒一次攻击，完美弹反率 50% 时
一个房间 3 只锈犬可攒 75-90 能量，接近一颗光核。→ [[策划案_能量系统]]

**与天赋系统**：弹反流的最佳练习对象（窗口大），同时成群时是普攻流的
能量提款机（"普攻命中回能量"面对弱怪时快速攒满）。→ [[策划案_天赋系统]]

**与怪物生态位**：近战追击型基准线。→ [[策划案_巡墙甲]] [[策划案_甲鼠]]

## 6. 美术/音效需求

需要什么精灵、动画帧、音效。
P0 阶段写占位需求，P2 做正式美术时链接到 → [[策划案_美术规范]]

**示例（锈犬）**：
精灵：俯视角犬类角色，比玩家略小，带锈蚀金属甲片质感
动画帧：idle（2 帧循环）、run（4 帧）、蓄力后蹲（2 帧）、扑击（3 帧）、硬直弹开（2 帧）、死亡（3 帧）
音效：蓄力低吼（1 个）、扑击风声（1 个）、被弹反金属碰撞声（1 个）、死亡消散（1 个）

## 7. 程序需求

场景树结构、Resource 字段定义、碰撞层需求、信号定义、关键技术方案。
技术方案遵循 Godot 设计哲学（详见 [[INDEX_开发路线图#一、Godot 设计哲学速查]]）：
- 场景结构采用组合模式（组件挂载），不用深层继承
- 数据配置使用 Custom Resource（.tres），不在代码中写死数值
- 节点间通信使用信号（Signal）解耦，向上信号、向下调用

**程序需求写作边界**

本章节写“实现意图、结构边界和验收约束”，不写未经验证的具体 Godot API 细节。

应该写：
- 需要哪些节点、组件、Resource、信号和碰撞层
- 哪些系统负责存数据、发事件、接收事件
- 哪些判定区在什么状态下有效
- 关键规则与可测试验收条件

不建议写死：
- 具体用哪个 Godot 属性开关控制判定
- 具体 API 调用顺序
- 未经验证的引擎行为假设
- 具体脚本内部 if / timer / signal 连接写法

若实现依赖 Godot 物理、Area2D、CollisionLayer/Mask、AnimationPlayer、Timer、Resource 导入、Autoload 初始化等引擎细节，开发前必须查官方文档或做最小复现验证，再接入主工程。

**示例（锈犬）**：

### 场景结构

锈犬场景基于 CharacterBody2D，采用组件组合模式：
- 状态机组件：巡逻 / 追击 / 蓄力 / 扑击 / 硬直 / 死亡
- HealthComponent：血量管理，受伤/死亡信号
- HitboxComponent：扑击时激活，扑击结束关闭
- HurtboxComponent：受击判定区域
- 占位 Sprite2D + AnimationPlayer

### 数据结构

使用 EnemyData Resource，字段：

| 字段 | 类型 | 值 | 说明 |
|------|------|-----|------|
| display_name | StringName | ENEMY_RUST_HOUND | 多语言 key |
| max_health | int | 40 | |
| attack_damage | int | 8 | |
| attack_interval | float | 2.0 | 攻击间隔（秒） |
| move_speed | float | 120.0 | |
| detection_range | float | 200.0 | 发现玩家距离 |
| attack_range | float | 50.0 | 进入攻击距离 |
| parry_window_frames | int | 16 | |
| stun_duration | float | 1.0 | 被弹反后硬直 |
| knockback_force | float | 60.0 | 弹反弹开距离 |

### 碰撞层需求

| Layer | 用途 |
|-------|------|
| Layer 1 | 环境（墙壁） |
| Layer 2 | 玩家 |
| Layer 3 | 敌人 |
| Layer 4 | 玩家 Hitbox |
| Layer 5 | 敌人 Hitbox |
| Layer 6 | 玩家 ParryArea |

锈犬 Body 在 Layer 3，检测 Layer 1（墙壁碰撞）。
锈犬 Hitbox 在 Layer 5，被 Layer 6（ParryArea）检测实现弹反判定。
锈犬 Hurtbox 在 Layer 3，被 Layer 4（玩家 Hitbox）检测实现受伤。

### 信号需求

- 扑击 Hitbox 激活时标记 `is_parryable = true`，供弹反系统判定
- 被弹反时通过 EventBus 发射 `parry_succeeded` 信号
- 死亡时通过 EventBus 发射 `enemy_died` 信号

## 8. 验收标准

- [ ] 具体可测试的条件
- [ ] ...

**示例（锈犬）**：
- [ ] 锈犬在检测范围内发现玩家后追击，进入攻击范围后执行扑击
- [ ] 扑击前摇可见，弹反窗口内按弹反键触发弹反成功
- [ ] 弹反成功后锈犬弹开并进入硬直，玩家获得能量
- [ ] 锈犬血量归零后播放死亡动画并消失
```

## 使用规则

1. 所有策划案命名格式：`策划案_名称.md`
2. 顶部写关联文档和路线图位置，能关联的就关联，没有的不硬凑
3. 策划案之间有关系的通过双链互相引用
4. 设计部分（1-6）侧重玩法与体验，程序需求（7-8）侧重实现约束
5. 不是每个章节都必须写——没有的跳过
