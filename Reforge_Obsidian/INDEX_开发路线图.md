## 一、Godot 设计哲学速查

### 1.1 场景 = 可复用模块

Godot 里一切皆场景。Player 是一个场景，Enemy 是一个场景，HUD 是一个场景，一个房间也是一个场景。场景可以嵌套：Room 场景里放 Enemy 场景和 CrystallizationPoint 场景。

**实操原则**：如果一个东西会被重复使用，或者逻辑上是独立的，就做成独立场景（`.tscn` 文件）。

### 1.2 组合优于继承

不要搞深层继承链（BaseEnemy → MeleeEnemy → RustHound）。用节点组合：一个 Enemy 场景里挂 HealthComponent、HitboxComponent、StateMachine 等子节点，每个组件是独立脚本。换一种怪就换组件组合。

**实操原则**：组件脚本放 `res://components/` 目录，谁需要谁挂上去。

### 1.3 信号（Signal）做松耦合

节点之间不直接调用方法，用信号通信。玩家弹反成功 → 发射 `parry_succeeded` 信号 → EnergySystem 监听到了加能量，HUD 监听到了播动画，敌人监听到了进入硬直。谁都不需要知道对方存在。

**实操原则**：向上通信用信号（Signal），向下通信用方法调用（call down）。同级之间如果需要通信，通过共同的父节点或 Autoload 中转。

### 1.4 Resource 做数据驱动

所有数值配置用自定义 Resource（`.tres` 文件）。怪物属性、天赋定义、能量参数全部是 Resource。GDScript 里定义 `class_name XXXData extends Resource`，编辑器里直接可视化编辑。

**实操原则**：代码里不写死数值。所有可调参数都在 Resource 里，方便迭代和 AI 批量生成配置。

### 1.5 Autoload 做全局管理

跨场景需要持续存在的管理器用 Autoload（单例）。比如 GameManager、EventBus、RunData（当局数据）。不要滥用——只有真正需要全局访问的才 Autoload。

### 1.6 多语言从第一天开始

所有面向玩家的文本用 `tr("KEY")` 调用。翻译文件用 CSV 格式，放 `res://localization/` 目录。初期写中文/英文/日文三语，Key 结构第一天就定好，后续加语言零成本。注意：世界观有双层命名切换系统（详见 [[DESIGN_世界观设计文档#3.2 双层命名切换系统]]），同一个机制有前期/后期两套名称，需要分别建 key（如 `HUD_ENERGY_EARLY` / `HUD_ENERGY_LATE`）。

## 二、项目目录结构（Godot 工程）

```
res://
├── autoloads/              # 全局单例脚本
│   ├── game_manager.gd     # 游戏状态管理
│   ├── event_bus.gd        # 全局信号总线
│   └── run_data.gd         # 当局运行数据（能量、光核、天赋）
├── components/             # 可复用组件场景
│   ├── health_component/
│   ├── hitbox_component/
│   ├── hurtbox_component/
│   ├── state_machine/
│   └── ...
├── data/                   # Resource 数据文件
│   ├── player/
│   ├── enemies/
│   ├── talents/
│   └── balance/            # 全局数值配置
├── entities/               # 游戏实体场景
│   ├── player/
│   ├── enemies/
│   └── interactables/      # 结晶点、祭坛等
├── ui/                     # UI 场景
│   ├── hud/
│   ├── talent_select/
│   └── menus/
├── rooms/                  # 房间场景
│   └── combat_room/
├── art/                    # 美术资源（精灵、Tile、图标）
│   ├── player/
│   ├── enemies/
│   ├── environment/
│   └── ui/
├── vfx/                    # 视觉特效场景
├── sfx/                    # 音效文件
├── music/                  # 音乐文件
├── localization/           # 多语言 CSV
│   └── translations.csv
└── project.godot
```

注意：这是 **Godot 工程内部**的目录结构，和 Obsidian Vault 是两个独立的东西。Godot 工程放你的代码项目文件夹，Obsidian Vault 放你的开发文档。

| 优先级 | 含义 | 目标 | 美术/音效/手感定位 |

| **P0** | 最小可玩闭环 | 一个房间能跑通核心循环 | 公开占位素材 + 最基础的打击反馈（能判断手感） |

## 七、关键架构决策备忘

> 技术方案的详细设计写在对应策划案的"程序需求"章节中，这里仅做索引和摘要。

### 7.1 弹反判定方案
> 详见 → [[策划案_玩家行为系统]]

- 按键触发 → Parry 状态 → ParryArea（Area2D）激活 → Timer 或时间累计控制有效窗口
- 前 0.15 秒为完美弹反（能量 10-12），0.15-0.30 秒为普通弹反（能量 5-6）
- 实现以秒为准，不依赖渲染帧率；帧数只作为 60fps 下的手感换算参考

### 7.2 状态机方案
> 详见 → [[策划案_玩家行为系统]]

- 节点式状态机：每个状态是 Node 子节点 + 独立脚本
- StateMachine 管理 `current_state`，每个状态有 `enter()` / `exit()` / `update()` / `physics_update()`
- 社区主流做法，GDQuest 推广的标准模式

### 7.3 天赋效果系统
> 详见 → [[策划案_天赋系统]]

- TalentData Resource 定义效果类型和参数
- TalentManager 节点持有已选天赋列表
- 各系统在关键时刻查询 TalentManager
- 一级天赋用属性修改器，二三级天赋用信号 + 条件触发

### 7.4 数据驱动方案
> 详见 → [[策划案_项目基建]]

- 所有配置用 Custom Resource（`.tres`）
- 编辑器内可视化编辑，代码里 `preload()` 加载
- 不用数据库，不用 JSON（存档除外）

### 7.5 能量系统架构
> 详见 → [[策划案_能量系统]]

- Autoload RunData 持有当局能量和光核数据
- EnergySystem 挂在 Player 上，通过信号广播变化
