# 开发日志

## 2026-05-15 | P0-1 项目基建

### 全局管理器
EventBus 是消息站，RunData 是当局账本。EventBus 负责广播“发生了什么”，RunData 负责保存“这一局的数据是多少”。

把事件通知和运行数据分开，可以避免玩家、敌人、HUD、能量系统互相直接调用，后续系统更容易扩展和调试。

心得：先分清“消息”和“数据”，再写系统。消息用于通知变化，数据用于记录状态。

### 多语言 Key 前置
玩家可见文本不直接写死在 UI 或脚本里，而是统一使用 key，例如 `UI_START`、`HUD_HEALTH`、`ENEMY_RUST_HOUND`。CSV 是人工维护的源表，Godot 会导入生成各语言的 `.translation` 运行资源。

多语言 key 不是只为翻译服务，而是把文本变成可管理资产。若后期再补 key，文本会散落在 UI、脚本、道具说明、天赋描述、剧情文本等地方，排查和替换成本很高。

Reforge 还有自己的双层命名系统：同一个机制在前期使用主角认知下的名称，后期随着真相揭露切换为真实名称。因此 key 还承担“叙事阶段切换接口”的作用。从 P0 开始使用 key，可以避免后期大拆 UI 和文本逻辑。

心得：早期做 key 很便宜，晚期补 key 很痛。文本没有入口，后期就是全项目考古。

## 2026-05-18 | P0-2 玩家行为系统

### 状态机与碰撞层
StateMachine 是管理玩家当前动作状态的调度器。它保证同一时间只有一个主要状态在工作，例如 Idle、Move、Attack、Parry、Dodge、Stun 或 Dead，避免所有输入和动作判断都堆在 `Player.gd` 里。

Godot 的碰撞 Layer / Mask 可以理解为：Layer 是“我属于哪一类碰撞”，Mask 是“我要检测哪一类碰撞”。项目中给数字层命名后，Layer 4 就是 PlayerHitbox，Layer 3 就是 EnemyBodyHurtbox；数字仍是底层实现，名字是为了让设计和调试更可读。

心得：状态机解决“现在该由谁处理输入”，Layer / Mask 解决“谁能碰到谁”。

## 2026-05-20 | P0-3 能量系统

### 超频完美弹反的层级流
一次超频完美弹反可以用来理解 P0-3 的完整分层：`PlayerParryState` 先发起 `overclock_action_requested("parry")`，表示玩家申请让这次弹反进入超频候选；`EventBus` 只负责广播这条消息；`OverclockSystem` 查询 `BalanceData`，知道超频弹反成本是 15，再查询 `RunData` 当前能量是否足够。若足够，`OverclockSystem` 记录 `accepted_actions["parry"] = 15`，并发出 `overclock_action_accepted("parry", 15)`。

当玩家弹反成功时，`Player` 发出 `parry_succeeded(true)`。这时 `EnergySystem` 先根据 `BalanceData` 的完美弹反回能值调用 `RunData.add_energy(12)`；随后 `OverclockSystem` 再调用 `RunData.consume_energy(15)`，扣费成功后发出 `overclock_action_triggered("parry", 15)`。HUD 只监听能量变化并刷新显示，Player 表现层只监听 `triggered` 做短暂金色占位反馈。

心得：`requested` 是申请，`accepted` 是批准候选，`triggered` 才是实际扣费成功；`BalanceData` 是价目表，`RunData` 是账本，`EnergySystem` 管基础收入，`OverclockSystem` 管超频消费。

### P0 后架构复盘点
当前为了快速跑通 P0，把 `EnergySystem` 和 `OverclockSystem` 暂时作为 Player 子节点挂载。这样做的好处是接入玩家动作信号快，测试成本低；风险是战斗规则系统会逐渐藏进 Player 场景里，后续如果房间、敌人、HUD、天赋都要参与结算，Player 会变得过重。

P0 结束后需要专门复盘：哪些节点属于玩家身体和动作，应该继续留在 Player；哪些节点属于战斗规则结算，应该移动到房间层的 `CombatSystems`。可考虑的结构是 `CombatRoom -> Player / Enemies / CombatSystems / HUD`，其中 `EnergySystem`、`OverclockSystem` 归入 `CombatSystems`。

心得：P0 可以先为“跑通闭环”让架构略微贴近 Player，但要明确这是阶段性选择；属于角色身体的留在 Player，属于战斗规则的 P0 后再讨论是否上移到房间层。

### 超频闪避兜底机制
当前 P0-3 先采用“超频闪避触发即扣 8 能量”。这个规则实现清楚，也符合闪避收益在动作开始时就发生的直觉；但它可能让玩家因为担心浪费能量而不愿使用超频闪避。

P0-4 敌人攻击与玩家受击判定接入后，需要复盘是否给超频闪避增加兜底机制。候选方向包括：闪避期间成功规避一次原本会命中的攻击才扣能；或者先扣能，若没有规避攻击则返还部分能量。当前不立刻修改，是因为没有敌人攻击命中链路时，系统无法可靠判断一次闪避到底是“失败”还是“成功躲开了危险”。

心得：弹反失败容易定义，闪避失败必须等敌人攻击判定出现后再定义。

## 2026-05-21 | P0-4 第一个敌人：锈犬

### 敌人 AI 状态机
锈犬的基础 AI 按状态拆分：Idle 负责待机与检测玩家，Chase 负责朝玩家移动和脱战返回。后续 Charge、Lunge、Recovery、ParriedStun、Dead 也会各自处理当前阶段的规则。

这样做的好处是每个阶段的触发条件、持续行为和退出条件都清楚；以后调试“为什么锈犬没有追击”或“为什么扑击没有结束”时，可以先看当前状态，再看该状态的入口和出口。

心得：AI 行为按状态拆分，每个状态只管当前阶段该做什么。

### 锈犬死亡链路
锈犬血量归零后，会进入 DeadState，关闭身体碰撞和攻击判定，并发出 enemy_died 事件给后续房间系统统计清怪。

真正完成死亡的是进入死亡状态、关闭判定、发清房事件。这样可以避免死后继续攻击、继续被命中，或被房间系统重复计入清怪进度。

### 节点顺序影响同帧判定优先级
锈犬验收测试发现，同一物理帧里如果玩家弹反和锈犬命中同时成立，节点执行顺序会影响结果。当前测试场景需要保持 Player 在 RustHound 前面，保证玩家弹反先关闭锈犬 AttackHitbox。

同帧战斗判定不能长期依赖节点顺序。后续需要加入明确的弹反优先级规则，让“弹反成功优先于本帧受伤”成为系统规则，而不是场景树顺序带来的结果。

## 2026-05-25 | P0-6 天赋系统

### 祭坛与天赋选择的职责分离
祭坛只负责消耗光核并发出 `talent_selection_requested`。`TalentSystem` 负责生成候选、接收选择、写入 `RunData.acquired_talents`，`TalentSelectionUI` 负责显示三张卡和接收玩家点击。

这样拆分后，祭坛是天赋选择入口，天赋系统是规则层，UI 是表现层。后续事件奖励、Boss 奖励、商店购买等来源也可以复用同一套天赋选择流程，只需要发出请求，不需要各自维护一套三选一逻辑。

### Godot 节点延迟删除
`queue_free()` 用于删除节点，但它是排队删除。调用后，Godot 会先把节点标记为等待释放，通常到当前帧结束或安全时机才真正移出场景树。被删除节点下面的子节点也会一起删除。

测试中如果调用 `queue_free()` 后立刻继续下一段逻辑，旧节点和子节点可能还在这一帧里接收信号。例如 Player 被排队删除时，子节点 `OverclockSystem` 可能还连着 EventBus，导致后续测试事件被重复处理。

需要确认节点已经离开场景树时，可以在 `queue_free()` 后等待一帧：`await get_tree().process_frame`。

### 天赋拥有权与运行时状态
天赋系统里需要区分“拥有权”和“运行时状态”。`RunData.acquired_talents` 记录本局已经获得哪些天赋；`TalentSystem` 记录这些天赋在当前战斗中留下的临时状态，例如敌人身上的裂纹层数、玩家当前护势层数、正在显示的三选一候选。

新一局开始时，清空已获得天赋还不够，也要清空对应的运行时状态。否则会出现玩家已经没有某个天赋，但该天赋留下的裂纹、护势或候选数据仍然残留的问题。

当前做法是 `RunData.reset_run()` 发出 `run_reset`，`TalentSystem.reset_runtime_state()` 清空 `active_options`、`crack_stacks` 和 `guard_momentum_stacks`。后续新增 Buff 或临时状态时，都要同步定义它的获得、消耗和清空时机。
