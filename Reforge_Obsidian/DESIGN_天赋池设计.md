## 一、文档定位

本文件用于规定 Reforge 天赋池的设计、记录和维护方式。它回答“一个天赋条目应该如何被设计、归档、筛选、迭代”。

天赋池设计文档服务三个目标：

- 让每个天赋都有清晰的构筑职责。
- 让天赋池可以长期扩展而不失控。
- 让后续 `DATA_天赋池` 中的条目具备统一格式，便于筛选、实现、测试和作品集展示。

本文件不记录完整天赋清单。具体天赋条目进入 [[DATA_天赋池]]。

---

## 二、天赋内容结构

每个天赋同时拥有玩家可见信息和内部维护信息。玩家可见信息负责快速理解，内部维护信息负责长期设计、实现和调整。

### 2.1 玩家可见信息

| 内容 | 作用 |
|------|------|
| 天赋名 | 提供记忆点和世界观味道 |
| 图标 / 插画 | 快速识别动作、关键词或流派方向 |
| 层级 | 表示一级 / 二级 / 三级 |
| 动作归属 | 表示弹反 / 普攻 / 闪避 / 超频 / 技能 / 中性 |
| 关键词 | 用短词压缩可复用机制 |
| 效果文本 | 描述这张天赋实际发生什么 |
| 数值 | 标注伤害、层数、持续时间、能量比例等 |
| Tooltip | 解释关键词、状态和特殊规则 |

卡面信息需要先让玩家知道“这张天赋会让自己怎么打”，再让玩家理解具体数值。关键词和 Tooltip 共同承担规则解释职责。

### 2.2 内部维护字段

每个天赋条目应使用固定字段记录，避免只写一个名字和效果就进入池子。

| 字段 | 说明 |
|------|------|
| 天赋 ID | 稳定英文标识，用于数据与代码引用 |
| 显示名称 | 面向玩家的名称，可随叙事阶段切换 |
| 层级 | 一级 / 二级 / 三级 |
| 主流派 | 弹反 / 普攻 / 闪避 / 技能 / 超频 / 中性 |
| 副流派 | 可选，用于桥梁天赋或混搭天赋 |
| 关键词 | 这个天赋使用或创造的关键词 |
| 机制类型 | 入口 / 引擎 / 消耗 / 桥梁 / 稳定性 / 核心 |
| 关联动作 | 这个天赋直接影响哪些动作 |
| 触发条件 | 什么时候生效 |
| 效果描述 | 生效后发生什么 |
| 代价 | 消耗、限制、风险或前置条件 |
| 组合对象 | 它希望和哪些天赋或动作形成组合 |
| 设计意图 | 为什么这个天赋应该存在 |
| 风险点 | 可能带来的平衡、理解或体验问题 |
| 当前状态 | 候选 / 已确认 / 已实现 / 待测试 / 调整中 / 废弃 |
| 备注 | 记录临时问题、待讨论点或版本说明 |

字段不要求每次都写成长段。低风险天赋可以简写，但“触发条件、效果、代价、设计意图”四项必须明确。

## 三、天赋关键词系统

天赋池优先围绕关键词扩展。关键词是被多张天赋复用的机制术语，用于压缩规则、降低读卡成本，并让同类天赋形成体系。

关键词需要对应明确机制，并能在多张天赋中反复出现。

关键词的作用：

- 压缩效果文本，减少重复解释。
- 降低玩家理解成本。
- 让相关天赋形成可维护的小体系。
- 让平衡调整拥有统一抓手。
- 避免天赋池变成一次性效果堆叠。

关键词设计标准：

- 每个关键词必须对应明确机制。
- 每个关键词至少能支撑多张天赋。
- 每个关键词应能通过 Tooltip 解释。
- 关键词名称应贴合 Reforge 的锻造、异力、结晶、过载等世界观语言。
- 关键词应服务动作、能量、状态或循环，避免只表达强弱评价。

### 3.1 第一批候选关键词

| 关键词 | 机制方向 | 适合承载的天赋 |
|--------|----------|----------------|
| 共振 | 一个动作影响下一次动作 | 弹反后强化普攻、闪避后强化弹反、技能后强化超频 |
| 回流 | 能量消耗后按条件返还 | 超频击杀返能、技能命中返能、连续动作降低净消耗 |
| 裂纹 | 对敌人叠加可爆发标记 | 普攻叠层、弹反加深裂纹、满层爆发或增伤 |
| 过载 | 短时强化并附带代价 | 超频强化、短时爆发、结束后暂时降低收益 |
| 护势 | 通过动作获得短暂保护 | 弹反护盾、闪避保护、失败保护、关键动作容错 |

### 3.2 暂缓关键词

以下关键词暂不进入第一批正式关键词，保留为后续扩展候选：

| 关键词 | 暂缓原因 |
|--------|----------|
| 凝核 | 更偏战后资源转换，容易变成结晶效率或经济天赋 |
| 锻痕 | 与裂纹同为叠层机制，需进一步区分“敌人叠层”和“玩家自身叠层”后再决定 |

第一批关键词优先覆盖战斗中高频出现、能绑定动作、能支撑多张天赋的机制。战后经济型关键词等光核、祭坛和事件系统更成熟后再扩展。

## 四、新天赋准入标准

一个天赋进入候选池前，需要回答以下问题：

1. 它改变哪个动作、资源或循环？
2. 它服务哪条构筑路线？
3. 它和哪些已有天赋或动作组合？
4. 它的代价是什么？
5. 它使用或创造哪个关键词？
6. 它是否提供了新的决策，还是只让数值变大？
7. 它是否和属性强化房、商店、事件或 Meta 成长重叠？
8. 玩家能否在三选一界面中快速理解它的用途？

满足以下条件之一，才适合进入天赋池：

- 改变动作规则或动作之间的关系。
- 改变能量获取、能量消耗或超频收益。
- 改变技能的获得、触发、充能或循环方式。
- 连接两个动作路线，形成混搭空间。
- 提高 Build 循环稳定性。
- 定义一条核心玩法循环。

只提供底层属性成长的条目不进入天赋池。

## 五、归入其他系统的内容

以下内容默认归入属性、商店、事件或 Meta 成长系统：

| 内容 | 归属 |
|------|------|
| 最大生命值增加 | 属性强化房 / Meta 成长 |
| 移动速度增加 | 属性强化房 / Meta 成长 |
| 防御力增加 | 属性强化房 / 商店 / Meta 成长 |
| 房间间回复量提高 | Meta 成长 / 关卡奖励 |
| 光核存储上限增加 | Meta 成长 |
| 基础能量池上限增加 | Meta 成长或特殊中性规则，需要单独确认 |
| 单纯金币、商店折扣、消耗品 | 商店 / 事件系统 |

天赋可以影响这些属性，但需要通过动作、资源或构筑循环产生关系。

例如：

- “生命值 +20”属于属性强化，不进入天赋池。
- “每次完美弹反后获得短暂护盾”可以进入天赋池，因为它把生存能力绑定到弹反行为。
- “移动速度 +10%”属于属性强化。
- “闪避后 1 秒内移速提高，并使下一次普攻附带冲击波”可以进入天赋池，因为它连接闪避和普攻。

## 六、流派配额原则

天赋池扩展时，应先保证每条主流派都有基础结构，再增加花哨组合。

每条主流派建议维护以下骨架：

| 类型 | 建议数量 | 作用 |
|------|----------|------|
| 一级入口天赋 | 2-3 个 | 给玩家看见路线方向 |
| 二级改造天赋 | 2-3 个 | 改变动作规则或资源关系 |
| 三级核心天赋 | 1-2 个 | 定义该流派的成型循环 |
| 桥梁天赋 | 2-4 个 | 连接其他动作路线 |
| 稳定性天赋 | 1-2 个 | 解决循环卡手、输入负担或资源波动 |

配额用于检查池子是否偏科，不作为硬性数量要求。

当某条流派出现以下情况，需要补池：

- 只有入口天赋，没有改造天赋。
- 只有伤害强化，没有资源或动作关系。
- 只有主路线，没有桥梁。
- 核心天赋依赖条件过多，难以成型。
- 稳定性工具缺失，导致路线看起来强但实际断循环。

## 七、桥梁天赋维护

桥梁天赋的职责是连接两个动作路线。它需要通过动作因果关系连接两条路线，避免把两个独立效果简单拼在一起。

桥梁天赋记录时需要额外写清：

| 字段 | 说明 |
|------|------|
| 起点动作 | 触发或启动桥梁的动作 |
| 终点动作 | 被强化、改变或引导使用的动作 |
| 连接方式 | 起点如何影响终点 |
| 组合目的 | 这个连接希望玩家形成什么循环 |

桥梁天赋的常见结构：

```text
动作 A 成功 → 改变动作 B
动作 A 消耗资源 → 动作 B 返还资源
动作 A 创造状态 → 动作 B 消耗状态
动作 A 提供安全窗口 → 动作 B 提供输出回报
```

桥梁天赋设计检查：

- 玩家是否能一眼看出它连接哪两个动作。
- 起点动作和终点动作是否都重要。
- 这个连接是否能引导玩家改变操作顺序。
- 它是否真的制造混搭，而非只让一个动作附带额外伤害。

---

## 八、稳定性天赋维护

稳定性天赋负责让 Build 循环更顺。它们经常没有最高伤害，但能决定某条路线是否可持续。

稳定性天赋需要记录“解决的问题”：

| 问题类型 | 设计方向 |
|----------|----------|
| 能量输入不稳定 | 增加条件性回能、提高关键动作收益 |
| 能量消耗过重 | 降低特定动作消耗、提供返还 |
| 输入负担过高 | 提供锁定、自动强化、下一次动作标识 |
| 触发条件太苛刻 | 增加窗口、延长状态、允许失败保护 |
| 选择过散 | 提高同路线天赋出现概率或提供选择修正 |
| 核心循环启动慢 | 提供启动层数、预充能或首个触发奖励 |

稳定性天赋的描述需要让玩家理解它的价值。若玩家无法看出它为什么有用，该天赋在三选一中会被长期低估。

## 九、天赋池表格维护规则

`DATA_天赋池` 应作为具体天赋条目的长期维护表。每个天赋只保留一个主条目，后续修改在状态、备注或变更记录中体现，避免同一想法散落在多个文件。

推荐状态：

| 状态 | 含义 |
|------|------|
| 候选 | 想法可用，但尚未确认进入正式池 |
| 已确认 | 设计上成立，等待实现或测试 |
| 已实现 | 已进入 Godot 数据或代码 |
| 待测试 | 已实现但需要 playtest 验证 |
| 调整中 | 体验、数值或描述需要修改 |
| 废弃 | 不再使用，保留原因便于回溯 |

状态只用于天赋条目管理，不替代 [[INDEX_开发路线图]]。路线图仍然负责记录某个阶段要实现哪些天赋。

维护原则：

- 新天赋先进入“候选”，通过准入标准后改为“已确认”。
- 已确认天赋应补齐触发条件、效果、代价和设计意图。
- 已实现天赋应记录对应数据 ID。
- 废弃天赋保留一行，不直接删除，避免未来重复提出相同问题。
- 每次大规模调整天赋池时，优先检查流派配额和桥梁分布。

## 十一、设计风险与待验证问题

1. 天赋池持续扩展后，是否会出现条目膨胀，导致三选一质量下降。
2. 每条流派是否都有入口、改造、核心、桥梁和稳定性工具。
3. 桥梁天赋是否真正改变操作顺序，还是只把两个效果缝在一起。
4. 稳定性天赋是否在描述和体验上足够有吸引力。
5. 关键词是否足够清晰，能否支撑多张天赋复用。
6. 关键词是否贴合世界观，同时保持规则含义明确。
7. 强力组合是否带有足够代价，避免成为长期唯一答案。
8. 是否有过多纯数值天赋挤占动作规则型天赋空间。
9. 天赋池是否和属性强化房、商店、Meta 成长发生职责重叠。
10. 废弃条目是否保留了足够原因，方便后续避免重复设计。
