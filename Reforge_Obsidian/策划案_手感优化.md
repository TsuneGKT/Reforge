> 类型：P0 手感审查与修复（技术调优为主，含少量规则调整）。
> 来源：对 `entities/player/` 状态机、`data/player/default_player_stats.tres`、`systems/feedback_system.gd` 的逐项代码审查。
> 结论：架构（状态机 / 信号 / 反馈系统分离）无需改动；手感问题集中在**输入响应**、**动作可打断性**、**判定覆盖**三处，外加打击感细节。

---

## 1. 设计目的

P0 核心循环依赖"弹反获能"。玩家要先觉得**操作跟手、判定讲理**，才愿意反复尝试弹反、把能量系统跑起来。如果在学会弹反之前就被"按了没反应""翻滚了还挨打""砍一刀拔不出来"这类挫败劝退，核心循环在第一步就断裂。

本案目标：消除三类高频挫败感，让 P0 战斗手感达到可对外展示（作品集 Demo）的水平。不追求最终平衡，只解决"发涩"和"不讲理"。

不做这些修复会怎样：当前攻击后摇约 0.17 秒内无法做任何防御动作，玩家进攻一次就可能无解吃一套；闪避最后 0.05 秒无敌已经结束但人还在滑行，"翻滚了还被打"会让玩家不信任闪避。两者叠加，玩家会判断"这游戏不跟手/不公平"，而这恰恰发生在他还没体会到弹反乐趣之前。

---

## 2. 问题清单（审查结论）

按"对手感影响"从高到低排列。

### 第一档 · 操作发涩的根因

**P1 没有输入缓冲（input buffer）**
- 现状：每个状态当帧直接读 `Input.is_action_just_pressed`（见各 `player_*_state.gd`）。在攻击锁定（0.35s）、弹反窗口（0.3s）、闪避（0.2s）、硬直（0.25s）期间按下的动作键**全部丢失**。
- 表现：按了没反应、连段打不出来、动作衔接卡顿。这是动作游戏"不跟手"的头号元凶。

**P2 攻击后摇约 0.17 秒不可打断**
- 现状：攻击时间线 = 前摇 0.08 → 判定 0.10 → 锁定至 0.35（`attack_lock_duration`）。判定结束后约 0.17 秒内**不能移动 / 闪避 / 弹反 / 接刀**。
- 表现：进攻一次后无法防御，敌人接招只能硬吃。玩家不敢进攻，战斗发懵。

### 第二档 · 判定不讲理

**P3 闪避无敌帧短于闪避时长**
- 现状：闪避 0.2s，无敌仅前 0.15s（`dodge_invincible_duration` < `dodge_duration`）。最后 0.05s 人还在滑行但已可被命中。
- 表现："明明翻滚了还是被打"，玩家不信任闪避。

**P4 弹反失手原地僵直 0.3 秒**
- 现状：未弹中时把 0.3s 窗口走完才解锁，期间原地无防护，且不可打断。
- 表现：一次误按惩罚 0.3s 偏重；叠加无输入缓冲，弹反时机显得不宽容。

### 第三档 · 打击感 / 重量感

**P5 攻击完全定身**：攻击时 `velocity` 归零，无前冲，挥刀像扎根，缺重量。
**P6 命中敌人无震屏**：`_on_attack_hit_target` 只发 `hit_stop_light` + `flash_white`，没有震屏 → 自己砍中比弹反还轻。
**P7 自己受击无顿帧**：`_on_player_hit` 只发 `flash_red` + 震屏，缺一下短顿帧来卖"被击中"。
**P8 完美弹反震屏未加强**：`is_perfect` 已传入但 `screen_shake_controller` 未使用，完美/普通弹反震感相同，核心爽点没给足。

> 非问题（设计选择，记录备查）：移动为瞬时速度、无加减速。俯视角动作常见做法，精准但偏"硬"。是否要更多重量感由后续决定，不在本案范围。

## 3. 修复方案

### P1 · 输入缓冲

引入"最近一次动作输入 + 时间戳"的缓冲，可被行动状态消费。

```
[规则] 输入缓冲
记录：玩家按下 attack / dodge / parry 时，记下动作类型与按下时刻
有效期：约 0.12 秒（按下后超过则作废）
消费：当状态进入"可接收新动作"时刻时，若缓冲内有未过期动作，立即执行并清空缓冲
```

可接收时刻：
- 攻击：判定结束、进入后摇可取消窗口时
- 闪避：闪避结束回到 idle/move 时
- 弹反：窗口结束回到 idle/move 时
- idle / move：本就当帧响应，缓冲作为兜底

### P2 · 攻击后摇可打断（dodge / parry cancel）

```
[攻击] 中断优先级（从高到低）
[最高] 死亡 / 受击 → 取消攻击，进入对应状态（现状已支持）
[高]   判定结束后，按下闪避 → 取消后摇，进入闪避
[高]   判定结束后，按下弹反 → 取消后摇，进入弹反
[中]   后摇自然结束（0.35s）→ 回到 idle/move
```

要点：取消窗口从"判定结束"（约 0.18s）开始，到后摇自然结束（0.35s）。即判定一旦打完，防御动作可随时把后摇打断。进攻不再无解。

备选/叠加：将 `attack_lock_duration` 从 0.35 适度下调（见 §4），缩短无操作的纯后摇。

### P3 · 闪避无敌覆盖全程

P0 阶段令无敌覆盖整个闪避（甚至略长于位移），消除"滑行尾巴"被打。

```
[闪避] 无敌帧
现状：0.15s（< 闪避 0.2s）
目标：≥ 0.2s（覆盖全程）；如需起手保护可在位移前加 1-2 帧无敌
```

### P4 · 弹反失手可被打断

```
[弹反] 失手处理
现状：原地僵直走完 0.3s 窗口
目标：窗口期内允许"按下闪避 → 取消弹反进入闪避"，给误按一个逃生出口
可选：缩短失手后的剩余僵直
```

### P5–P8 · 打击感

- P5：攻击判定帧期间给一个 20–40px 的小前冲（沿朝向），增重量、补攻击距离。
- P6：`_on_attack_hit_target` 追加一次轻量震屏请求。
- P7：`_on_player_hit` 追加一次短顿帧（轻于完美弹反）。
- P8：`screen_shake_controller` 读取 `is_perfect`，完美弹反用更强 / 更长的震屏（新增 strong 档）。

## 4. 数值与依据

| 参数 | 现状 | 目标 | 依据 |
|------|------|------|------|
| 输入缓冲有效期 | 无 | ~0.12s | 覆盖一次动作衔接的预输入窗口；过长会"手放开了还触发" |
| 攻击取消窗口起点 | 不可取消 | 判定结束（~0.18s）起 | 判定打完即可被防御动作打断，进攻不再无解 |
| attack_lock_duration | 0.35s | 0.28–0.30s（待手感校准） | 缩短纯后摇；有取消窗口后可略缩，避免连段过快需配合攻速 |
| dodge_invincible_duration | 0.15s | 0.20s（=闪避时长） | 消除滑行尾巴被打 |
| 弹反失手可打断 | 否 | 是（闪避打断） | 给误按逃生出口，降低 0.3s 僵直的惩罚感 |
| 命中震屏 | 无 | light | 自己命中应有脆感，不弱于弹反 |
| 受击顿帧 | 无 | 轻（< 完美弹反） | 卖出"被击中"，提升可读性 |
| 完美弹反震屏 | 与普通相同 | strong（更强/更长） | 兑现"完美"爽点 |

> 数值均为起始值，需进游戏实测校准；本案不锁定最终平衡。

## 5. 程序需求

遵循 Godot 设计哲学（组合、Resource 驱动、信号解耦，详见 [[INDEX_开发路线图#一、Godot 设计哲学速查]]）。

### 输入缓冲
- 实现意图：一个轻量输入缓冲（记录最后动作 + 时间戳），可由玩家或状态机查询/消费。倾向放在 `Player` 或独立小组件，避免每个状态各写一套 `is_action_just_pressed`。
- 各动作状态在"可接收时刻"查询缓冲并消费，而非当帧直接读输入。
- 缓冲有效期作为可调参数（写入 `PlayerStats` 或缓冲组件导出字段，不写死）。

### 攻击取消
- `player_attack_state.gd` 在判定结束后进入"可取消"标记；其 `physics_update` 在该标记下检测（缓冲的）闪避 / 弹反并切换状态。
- 取消窗口起点与 `attack_lock_duration` 均为可调参数。

### 闪避无敌
- `dodge_invincible_duration` 调至 ≥ `dodge_duration`（改数据 `default_player_stats.tres`，逻辑 `player_dodge_state.gd` 已按此字段控制无敌，无需改结构）。

### 弹反失手打断
- `player_parry_state.gd` 在窗口期内增加对（缓冲的）闪避输入的检测，命中则切换到闪避。

### 打击感
- 数值字段（前冲距离）尽量放入 Resource。
- 反馈走现有 `FeedbackSystem` 信号：
  - `_on_attack_hit_target` 追加 `SCREEN_SHAKE_LIGHT`。
  - `_on_player_hit` 追加一档轻顿帧（可新增 `HIT_STOP_LIGHT` 调用或更轻档）。
  - `screen_shake_controller` 支持按 `is_perfect` 选择强度（新增 strong 参数）。

> 涉及 Engine.time_scale 顿帧、Camera2D 偏移震屏、Area2D 判定开关的引擎细节，按现有实现沿用；改动前若不确定先做最小验证。

## 6. 验收标准

- 在攻击锁定 / 闪避 / 弹反 / 硬直期间预输入的动作，状态解锁瞬间能被执行（输入缓冲生效）。
- 攻击判定结束后，可用闪避 / 弹反立即打断后摇，不必等满 0.35s。
- 整个闪避过程不被命中（无敌覆盖全程）。
- 弹反失手期间可用闪避脱出。
- 自己命中敌人有顿帧 + 轻震屏；自己受击有短顿帧 + 红闪 + 震屏。
- 完美弹反的震屏明显强于普通弹反。
- 上述改动不破坏现有玩家状态机测试（`tests/player_*`、`tests/p0_7_*`、`tests/overclock_*`）；新增行为补对应测试。

## 7. 与其他系统的联系

- **与能量系统**：手感顺滑直接提升弹反成功率与尝试意愿，是能量产出的前置条件。→ [[策划案_能量系统]]
- **与超频系统**：攻击 / 闪避 / 弹反进入时已 `request_overclock_action`，本案不改超频触发链路，仅改这些动作的响应与可打断性，需回归测试超频结算不受影响。
- **与敌人节奏**：闪避无敌与攻击可打断改变了玩家容错，锈犬等教学怪的窗口/间隔是否仍合适，实测后回看 → [[策划案_锈犬]]
